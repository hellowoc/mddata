# TCP 协议详解

## 文档主线

从你已经知道的出发——IP 层只负责"把包从 A 路由到 B"，不保证到达、不保证顺序、不保证不重复。TCP 在 IP 之上**用序号 + 确认 + 重传三个机制**，把不可靠的 IP 传输变成可靠的字节流。

这条主线贯穿全文：TCP 的每一个设计——三次握手、累积确认、快速重传、滑动窗口、四次挥手——都是为了解决 IP 层的一个具体缺陷。

---

## 第一章：TCP 在哪、干什么

### 1.1 回顾你已经知道的

IP 层做的事：源 IP → 目标 IP，查路由表，逐跳转发。但 IP 有三个致命问题：

1. **不保证到达**：路由器丢包（拥塞、TTL 耗尽），IP 层不通知你
2. **不保证顺序**：两个 IP 包可能走不同的路由路径，后发的先到
3. **不保证不重复**：路由器可能重复转发同一个包

对于 MQTT 这种应用来说，这不可接受——CONNECT 报文丢了，MQTT 连接永远建不起来。

### 1.2 TCP 的位置

```
应用层：MQTT 报文
────────────── ▲ TCP 给上层提供的接口：可靠的字节流
传输层：TCP    │  （不管上层数据是什么意思，只保证字节完整、有序到达）
────────────── │
网络层：IP     │  TCP 依赖 IP，但 IP 不可靠
────────────── │
```

TCP 对上层说："你给我一串字节，我保证它完整、有序、不重复地到达对方。"

TCP 对下层说："IP，你给我一个不可靠的传输服务，我用序号+确认+重传来弥补你的缺陷。"

### 1.3 核心数据结构

TCP 在内核中为每个连接维护一个结构体（两端各有一套）：

```c
struct tcp_sock {
    // === 发送侧 ===
    uint32_t snd_nxt;    // 下一个要发送的字节序号
    uint32_t snd_una;    // 最早未确认的字节序号（"una" = unacknowledged）
    uint32_t snd_wnd;    // 发送窗口大小（对方告诉我它还能收多少）

    // === 接收侧 ===
    uint32_t rcv_nxt;    // 期望收到的下一个字节序号
    uint32_t rcv_wnd;    // 接收窗口大小（我还能收多少）

    // === 重传队列 ===
    // 每个已发送但未确认的数据段都缓存在这里，超时则重发
};
```

**序号是理解 TCP 的关键**。TCP 把要发送的数据看成一个连续的字节流，每个字节有一个编号，从某个随机初始值开始。后续所有机制（确认、重传、窗口控制）都基于这个"字节序号"。

---

但 TCP 不能上来就发数据——通信双方必须先确认"对方存在、能正常收发"。所以有了三次握手。

## 第二章：三次握手 — 建立连接

### 2.1 为什么需要握手

IP 层是"无连接"的——你可以往任何 IP 地址发包，不需要对方同意。但 TCP 是"面向连接"的——双方必须先对通信参数达成一致（各自的起始序号、各自能收多大的数据），才能开始传输。

### 2.2 三次握手的每一步

```
客户端                                          服务器
  │                                               │
  │── SYN(seq=x) ────────────────────────────→  │
  │   标志位: SYN=1                              │
  │   序号: x (客户端的初始序号，随机生成)           │
  │   ACK号: 0 (SYN报文不确认任何数据)              │
  │                                               │
  │←── SYN+ACK(seq=y, ack=x+1) ───────────────│
  │   标志位: SYN=1, ACK=1                       │
  │   序号: y (服务器的初始序号，随机生成)           │
  │   ACK号: x+1 (确认收到了客户端的 SYN)           │
  │                                               │
  │── ACK(ack=y+1) ──────────────────────────→  │
  │   标志位: ACK=1                               │
  │   ACK号: y+1 (确认收到了服务器的 SYN)           │
```

### 2.3 每一步的精确含义

**第一步**：客户端说"我要建立连接，我的起始序号是 x"。SYN 标志位表示这是一个连接请求。x 是一个随机 32 位整数（不是从 0 开始——防御伪造 TCP 包攻击）。

**第二步**：服务器说"我收到了你的请求（ack=x+1 就是确认），我同意建立连接，我的起始序号是 y"。服务器把这个 SYN 和对客户端 SYN 的确认 **ACK 合并到一个包里**——理由是这两个动作之间没有等待，服务器收到 SYN 后立刻就能决定接受连接。

**第三步**：客户端说"我收到了你的确认"。此时客户端的 SYN 被确认了，服务器的 SYN 也被确认了——双方都确认"对方能收到我的消息"。

### 2.4 为什么是三次，不是两次也不是四次

如果只有两次（客户端 SYN → 服务器 SYN+ACK，结束）：

```
客户端说"我要连"，服务器说"好的"。服务器认为连接建立了，开始等数据。
但客户端可能根本没收到服务器的回应——网络丢包。
服务器在空等，客户端以为没连上，又发了一个 SYN。
```

两次的问题：服务器无法确认"客户端收到了我对它 SYN 的确认"。

如果是四次：

```
步骤2a: 服务器 → 客户端: ACK(ack=x+1)   "你的 SYN 收到了"
步骤2b: 服务器 → 客户端: SYN(seq=y)      "我的起始序号是 y"
```

TCP 把 2a 和 2b 合并到第二个包里（SYN+ACK），因为这两个动作之间没有任何等待——服务器收到 SYN 后当然愿意建立连接，ACK 和 SYN 可以同时发出。**所以三次是理论上的最小次数，不是随意定的。**

### 2.5 握手完成后

```
双方各有了：
  - 对方的初始序号（x 和 y）
  - 确认对方能正常收发

内核中 tcp_sock 的初始状态：
  客户端：snd_nxt = x+1, snd_una = x+1, rcv_nxt = y+1
  服务器：snd_nxt = y+1, snd_una = y+1, rcv_nxt = x+1
```

## 第三章：数据传输 — 序号、确认、累积确认

### 3.1 TCP 怎样传输数据

TCP 把应用层给的数据切分成若干**TCP 段（segment）**，每个段加上 TCP 头后交给 IP 层。一个典型的 TCP 段：

```
┌────────────┬──────────────────────────────────┐
│ TCP 头(20B) │  数据（应用层给的字节流的一部分）   │
│             │                                   │
│ seq=1000    │  "GET /api/status HTTP/1.1\r\n"  │
│ ack=5000    │  (假设这个段的长度是 100 字节)       │
│ len=100     │                                   │
└────────────┬──────────────────────────────────┘

含义：这个 TCP 段携带的字节是序号 1000-1099
     （seq=1000, len=100 → 覆盖的序号范围是 [1000, 1100)）
```

**序号是字节级别的**：如果你连续发了三个段，每个 100 字节：

```
段1: seq=1000, len=100  → 字节 1000-1099
段2: seq=1100, len=100  → 字节 1100-1199
段3: seq=1200, len=100  → 字节 1200-1299
```

### 3.2 ACK — 确认

接收方收到数据后，回复 ACK 段。ACK 号的含义是"序号 N 之前的字节我全收到了，下一个期望收到的字节是 N"。

```
接收方收到段1 后：回 ACK=1100  "1000-1099 收到了，下一个期望 1100"
接收方收到段2 后：回 ACK=1200  "1100-1199 也收到了，下一个期望 1200"
```

**TCP 用累积确认，不是逐段确认**：

```
如果接收方收到段1和段3，但段2丢了：
  收到段1 → 回 ACK=1100
  收到段3 → 回 ACK=1100  "我还是在等 1100，后面的（段3）收到了但不连续，先缓存"

发送方看到两次 ACK=1100 的重复确认，知道段2可能丢了。
```

### 3.3 从发送方视角看数据传输过程

```
时间 →

  发送方                             接收方
    │                                  │
    │── 段1 (seq=1000, len=100) ───→  │
    │── 段2 (seq=1100, len=100) ───→  │
    │── 段3 (seq=1200, len=100) ───→  │
    │                                  │
    │←── ACK=1200 ──────────────────│  "1000-1199 全收到了"
    │    (这意味着段1和段2都到了，      │
    │     段3还没到)                    │
    │                                  │
    │── 段3 重发(seq=1200, len=100)──→│  "段3丢了，重发"
    │←── ACK=1300 ──────────────────│  "1200-1299 到了"
```

---

累积确认解决了"确认数据到达"的问题。但网络会丢包——ACK 本身可能丢，数据段本身可能丢。TCP 怎么发现丢包、怎么重发？

---

## 第四章：重传机制

### 4.1 超时重传 — 基础机制

每发一个段，TCP 启动一个定时器。在定时器到期前收到 ACK → 取消定时器。定时器到期还没收到 ACK → 重发。

```
发送方发段(seq=1000)，启动定时器。
  │
  ├── 情况1：定时器到期前收到 ACK=1100 → 正常，继续发下一个段
  │
  └── 情况2：定时器到期，没收到 ACK → 重发 seq=1000
      每次重传，下一次的超时时间翻倍（指数退避），最多重试 15 次
```

超时时间（RTO, Retransmission Timeout）不是固定的——TCP 根据网络延迟动态计算。每次收到 ACK 时测量 RTT（Round-Trip Time，往返时间），用加权平均算法算出 RTO。

### 4.2 快速重传 — 优化机制

超时重传的缺点是慢——等一个 RTO 才能发现丢包。快速重传用**重复 ACK** 来提前发现丢包：

```
发送方连续发 5 个段：
  seq=1000, 1100, 1200, 1300, 1400

接收方收到 1000 → ACK=1100
接收方收到 1100 → ACK=1200
接收方没收到 1200（丢了）
接收方收到 1300 → ACK=1200  "我还是在等 1200"   ← 第1次重复ACK
接收方收到 1400 → ACK=1200  "我还是在等 1200"   ← 第2次重复ACK
                           "我还是在等 1200"   ← 第3次重复ACK

发送方收到 3 次相同的 ACK=1200 → 知道段3丢了 → 不等超时，立即重发 seq=1200
```

**为什么是 3 次重复 ACK 才触发快速重传？** 因为 1-2 次重复 ACK 可能是网络轻微抖动（段乱序到达），3 次基本能确认是真的丢了。

---

数据可靠传输的问题在上一章通过确认+重传解决了。但还有一个问题：**发送方发太快，接收方处理不过来，数据在接收方缓冲区溢出——TCP 怎么控制发送速度？**

## 第五章：流量控制 — 滑动窗口

### 5.1 问题

接收方的 TCP 接收缓冲区是有限的（内核分配的内存）。如果发送方无限制地发，接收方缓冲区满了，之后到达的段全部丢弃——发送方又要重传，浪费带宽。

### 5.2 窗口机制

接收方在每一条 ACK 中附带**窗口大小（Window Size）**，告诉发送方："我的缓冲区还剩 N 字节，你最多只能再发 N 字节"。

```
接收方：

  TCP 接收缓冲区（如 64KB）
  ┌────────────────────────────────────────────┐
  │ 已确认并交付应用 │ 已收到但未确认 │ 空闲空间   │
  │    (应用读了)    │   (等待中)    │ (窗口大小) │
  └────────────────────────────────────────────┘
                                          ↑
                              这个值放进 ACK 的 Window Size 字段
```

发送方维护一个**发送窗口**——已发送但未确认的字节总数不能超过接收方通告的窗口大小。窗口内的段可以连续发送，不用等上一个 ACK。

```
发送方视角：

  序号空间： [已确认] [已发送未确认] [可发送但未发送] [不能发送（窗口外）]
            └──────┘ └─────────────┘ └───────┬───────┘ └─────────────────┘
                      ↑                     ↑
                   snd_una              snd_nxt          snd_una + snd_wnd
```

这样就做到了**发送方发多快，完全由接收方决定**。

---

数据传完了，连接怎么关？TCP 是全双工的——双方可以独立决定"我不发数据了"。关闭需要四次交互。

---

## 第六章：四次挥手 — 断开连接

### 6.1 为什么挥手比握手多一次

握手时，服务器收到 SYN 后**立刻**就能决定接受连接——ACK 和 SYN 可以合并到一个包里（SYN+ACK）。所以三次。

挥手时，一方说"我不发了"，但另一方**可能还有数据没发完**。ACK（"知道了"）和 FIN（"我也不发了"）之间可能隔了很久——不能合并。所以四次。

### 6.2 每一步

```
主动关闭方                                        被动关闭方
    │                                               │
    │── FIN(seq=m) ────────────────────────────→  │
    │    "我不再发数据了（但我还能收）"                │
    │                                               │
    │←── ACK(ack=m+1) ──────────────────────────│
    │    "知道了，你那边关了"                         │
    │                                               │
    │    ★ 此时：主动方→被动方 的方向已关闭             │
    │    但 被动方→主动方 的方向还开着                  │
    │    被动方可能还有数据要发                         │
    │                                               │
    │←── FIN(seq=n) ────────────────────────────│
    │    "我的数据也发完了，我也关了"                    │
    │                                               │
    │── ACK(ack=n+1) ──────────────────────────→  │
    │    "知道了"                                    │
    │                                               │
    │    主动方等 2MSL（约 2 分钟）后彻底关闭           │
```

### 6.3 为什么主动关闭方最后要等 2MSL

MSL = Maximum Segment Lifetime，一个 TCP 段在网络中的最大存活时间（通常 30 秒到 2 分钟）。

主动方发完最后一个 ACK 后等 2MSL 的原因：

1. **如果最后的 ACK 丢了**——被动方超时后会重发 FIN。如果主动方已经关了，永远收不到重发的 FIN——被动方永远关不掉。在 2MSL 内，主动方还能收到这个重发的 FIN 并重新回 ACK。

2. **清理网络中的"僵尸"段**——等待 2MSL 保证这个连接的所有 TCP 段都从网络中消失，不会干扰下次用相同端口建立的新连接。

## 第七章：TCP 状态机全貌

### 7.1 11 种状态

```
                            ┌──────────────────────────────┐
                            │           CLOSED              │
                            └──────────┬───────────────────┘
                                       │
                        主动打开: SYN  │    被动打开: LISTEN
                                       │
                            ┌──────────▼───────────────────┐
                            │         SYN_SENT              │
                            └──────────┬───────────────────┘
                                       │ recv SYN+ACK, send ACK
                                       │
              ┌────────────────────────▼──────────────────────────┐
              │                   ESTABLISHED                      │
              │   数据传输阶段。send()/recv() 都在这个状态。          │
              └──────┬──────────────────────────────┬─────────────┘
                     │                              │
         主动关闭: send FIN                 被动关闭: recv FIN
                     │                              │
        ┌────────────▼──────────┐      ┌────────────▼──────────┐
        │     FIN_WAIT_1        │      │      CLOSE_WAIT        │
        └────────────┬──────────┘      └────────────┬──────────┘
                     │ recv ACK                     │ send FIN
        ┌────────────▼──────────┐      ┌────────────▼──────────┐
        │     FIN_WAIT_2        │      │      LAST_ACK          │
        └────────────┬──────────┘      └────────────┬──────────┘
                     │ recv FIN, send ACK           │ recv ACK
        ┌────────────▼──────────┐      ┌────────────▼──────────┐
        │      TIME_WAIT        │      │        CLOSED          │
        │    (等 2MSL)          │      └────────────────────────┘
        └────────────┬──────────┘
                     │ 2MSL 到期
        ┌────────────▼──────────┐
        │        CLOSED         │
        └───────────────────────┘
```

### 7.2 最重要的几个状态

| 状态 | 出现在什么时候 | 能做什么 |
|------|-------------|---------|
| ESTABLISHED | 三次握手完成后 | send/recv 正常收发数据 |
| FIN_WAIT_1 | 你主动调 close() | 等对方确认你的 FIN |
| FIN_WAIT_2 | 你的 FIN 被对方确认了 | 你还能收，等对方的 FIN |
| TIME_WAIT | 你收到对方的 FIN 并回了 ACK | 什么都不能做，等 2MSL |
| CLOSE_WAIT | 你收到对方的 FIN | 你可能还有数据要发，发完后你发 FIN |

## 第八章：TCP 在本工程中的使用

### 8.1 MQTT 底层用的就是 TCP

```
mqttsrv.cpp — mqttThread::run():

test->connect("cloud.chinaamd.cn", 1884, 50);
// mosquitto 内部调用：
//   socket(AF_INET, SOCK_STREAM, 0)     ← SOCK_STREAM = TCP
//   connect(sockfd, &server_addr, ...)   ← 三次握手在这里完成
//   （如果用了 TLS，在这之后 TLS 握手在 TCP 连接之上进行）
//   MQTT CONNECT 报文在 TCP/TLS 连接之上发出

while (1) {
    test->loop();
    // mosquitto 内部调用 select() 检查 socket 可读
    // 然后 recv() 读 TCP 字节流
    // 解析 MQTT 报文 → on_message() 回调
}
```

### 8.2 用 BSD Socket API 直接操作 TCP 的模板

基于本工程已有的 mynetwork.cpp（UDP）代码风格，下面是 TCP 客户端的最小实现模板：

```c
// ====== TCP 客户端（参照 mynetwork.cpp 的风格） ======

int tcp_connect(const char *server_ip, int port)
{
    // 1. 创建 TCP socket
    int fd = socket(AF_INET, SOCK_STREAM, 0);
    //                        ↑ TCP = SOCK_STREAM
    // 如果失败，fd = -1

    // 2. 设置 socket 选项
    int val = 1;
    setsockopt(fd, SOL_SOCKET, SO_REUSEADDR, &val, sizeof(val));

    // 3. 填充服务器地址
    struct sockaddr_in server_addr;
    memset(&server_addr, 0, sizeof(server_addr));
    server_addr.sin_family = AF_INET;
    server_addr.sin_port   = htons(port);
    inet_pton(AF_INET, server_ip, &server_addr.sin_addr);

    // 4. 连接服务器（三次握手）
    int ret = connect(fd, (struct sockaddr*)&server_addr, sizeof(server_addr));
    if (ret != 0) {
        close(fd);
        return -1;  // 连接失败
    }
    // connect 返回 0：三次握手完成，进入 ESTABLISHED 状态
    return fd;
}

int tcp_send(int fd, const void *data, int len)
{
    // send 保证：len 字节按序、完整的到达对方 TCP 缓冲区
    // 返回值：实际发送的字节数，可能 < len（如果发送缓冲区暂时满了）
    return send(fd, data, len, 0);
}

int tcp_recv(int fd, void *buf, int buf_size)
{
    // recv 阻塞等待数据到达
    // 返回值：实际收到的字节数
    // 注意：TCP 是字节流，可能你 send("hello") 两次，
    // 对方一次 recv 收到 "hellohello"——必须自己在字节流之上定义消息边界
    return recv(fd, buf, buf_size, 0);
}

void tcp_close(int fd)
{
    // close 触发四次挥手
    close(fd);
}
```

### 8.3 TCP vs UDP — 本工程的选型逻辑

本工程三通道全用 UDP 而非 TCP，现在你理解了 TCP 之后，可以看出原因：

| | TCP | UDP（本工程用法） |
|------|-----|-----|
| 连接 | 必须先 connect 建立连接 | 无连接，直接 sendto |
| 可靠性 | 自动重传丢的字节 | 不保证，应用层自建 CRC + 序列号重传 |
| 延迟 | 丢包时 TCP 阻塞等重传 → 不可预测 | 丢包不影响后续包的收发 |
| 适用 | MQTT（少量命令，可靠性优先） | FPGA 相机（每秒数千帧，延迟优先） |

工业相机每秒数千帧——丢一帧不影响，但 TCP 等待重传引入的延迟会导致喷阀错过物料。所以用 UDP + 应用层自定义可靠性——丢帧直接丢弃等下一帧，不阻塞。

而对于 MQTT——每秒几条命令，可靠性优先——TCP 是正确选择。

## 文档总结

这条主线从第一章到第八章没有变过：**TCP 用序号 + 确认 + 重传，把不可靠的 IP 变成可靠的字节流**。

- 第一、二章：建立连接（三次握手），双方交换初始序号
- 第三、四章：数据传输——序号标记字节、ACK 累积确认、超时/快速重传补充丢失
- 第五章：流量控制——接收方用窗口限制发送方速度
- 第六、七章：断开连接（四次挥手），状态机全貌
- 第八章：回到本工程代码，看 TCP 在 socket API 层面怎么调用
