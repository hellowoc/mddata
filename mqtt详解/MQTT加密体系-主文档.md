# MQTT 加密体系 — 框架地图

## 0. 先看问题：没有加密时，MQTT 长什么样

```
工控机 ──── WiFi/以太网 ──── cloud.chinaamd.cn:1883

工控机发出的数据：
  "{"cmdCode":"CMD_OpenSwitch","content":"1:0:1:0","devCode":"ZKGDDEV47"}"
  
任何人在网络上抓包（tcpdump、Wireshark），直接看到上面的 JSON 明文。
如果有人劫持了路由器，他不仅能看，还能伪造命令发给设备。
```

加密要解决两个问题：
- **A. 防偷看（机密性）**：第三者截获了数据包，看不懂内容
- **B. 防冒充（身份认证）**：第三者伪造不了服务器的身份

---

## 1. 第一层：对称加密 — "同一把钥匙"

```
加密：明文 + 密钥 → 加密算法(AES) → 密文（乱码）
解密：密文 + 密钥 → 解密算法(AES) → 明文

同一把密钥既用来加密也用来解密 → "对称"
```

**优点**：速度快。加密 1MB 数据几乎不耗 CPU。

**致命缺陷**：密钥怎么传给对方？如果通过网络传密钥，网络本身就是不安全的——截获了密钥等于白加密。如果当面传，那跟当面传数据没区别。

**在 MQTT 中**：TLS 握手完成后，后续所有 MQTT 报文用对称加密传输。密钥是握手过程中协商出来的，不需要通过网络明文传。

---

## 2. 第二层：非对称加密 — "两把不同的钥匙"

```
公钥（公开的） + 私钥（保密的）= 一对

用公钥加密 → 只有私钥能解密
用私钥加密 → 只有公钥能解密（这就是"数字签名"的数学基础）
```

**优点**：公钥可以公开传到任何地方，不怕截获。因为截获了公钥也没用——解不了密，只有私钥持有者才能解密。

**缺点**：速度慢。比对称加密慢 100-1000 倍。不适合加密大量数据。

**在 MQTT 中**：TLS 握手阶段用非对称加密。客户端用服务器的公钥加密一个"临时密钥"，然后双方用这个临时密钥切换到对称加密。非对称加密只用来安全地传递"对称密钥"。

---

## 3. TLS 是怎么把两者结合起来的

TLS = Transport Layer Security。它的核心思路：

```
第一阶段：非对称加密（慢但安全）
  → 只用来安全地传输"对称密钥"
  
第二阶段：对称加密（快）
  → 用协商好的对称密钥加密所有后续数据
```

**TLS 握手（简化版）**：

```
步骤1: 客户端 → 服务器: "你好，我支持 AES、RSA、TLSv1"
步骤2: 服务器 → 客户端: "好，我们用 AES + RSA" + 服务器证书
步骤3: 客户端: 验证证书（确保证书是真的，服务器不是冒充的）
步骤4: 客户端: 生成一个"会话密钥"（对称密钥）
步骤5: 客户端 → 服务器: 用服务器公钥加密"会话密钥"，发给服务器
步骤6: 服务器: 用私钥解密，拿到"会话密钥"
步骤7: 之后双方用"会话密钥"做对称加密通信
```

步骤 1-6 就是**非对称加密阶段**（慢，但只做一次），步骤 7 开始是**对称加密阶段**（快，后续所有数据都用这个）。

---

## 4. 证书 — "身份证"

上面步骤 3 说"验证证书"。问题来了：服务器发来一个"证书"说自己是 cloud.chinaamd.cn，客户端怎么知道这个证书是真的？

```
证书 = 服务器的"身份证"
证书包含：
  - 服务器域名（cloud.chinaamd.cn）
  - 服务器的公钥
  - 颁发者（CA，Certificate Authority）的签名
```

**CA（证书颁发机构）** 是公认的第三方信任机构。

验证逻辑：
```
1. 客户端内置了 CA 的根证书（root.crt）→ 这是信任的"根"
2. 服务器发来自己的证书
3. 客户端用 root.crt 里的 CA 公钥，验证服务器证书上的 CA 签名
4. 签名验证通过 → 证书确实是 CA 颁发的 → 服务器身份可信
5. 从证书中取出服务器的公钥 → 用于后续加密
```

这就是 `tls_set("./root.crt", NULL)` 做的事——加载这个"根信任"文件。

**`tls_insecure_set(true)` 跳过的是什么**：
证书上写的是 `cloud.chinaamd.cn`，但设备可能通过 IP 地址连接。严格模式下，TLS 会检查"证书上的域名 == 你实际连接的地址"，不匹配就拒绝。`tls_insecure_set(true)` 跳过了这个检查——信任证书，但不检查域名。

## 5. 这些概念在本工程代码中的位置

```
mqttsrv.cpp:93-95
┌─────────────────────────────────────────────────────┐
│ test->tls_opts_set(1, "tlsv1", NULL);              │  ← 选择 TLS 协议版本
│ test->tls_insecure_set(true);                       │  ← 跳过域名验证
│ test->tls_set("./root.crt", NULL, NULL, NULL, NULL);│  ← 加载 CA 根证书
└─────────────────────────────────────────────────────┘

这些调用告诉 mosquitto 库：
1. 用 TLSv1 版本做加密
2. 验证服务器证书（cert_reqs=1 = SSL_VERIFY_PEER）
3. 用 ./root.crt 作为信任根
4. 但不验证证书域名是否匹配

之后 mosquitto 内部（通过 OpenSSL）自动完成：
- TCP 连接建立后 → TLS 握手 → 加密通道建立
- test->loop() 内部 select+recv → 收到的数据 OpenSSL 自动解密 → 得到明文 MQTT 报文
- test->publish() → 明文 MQTT 报文 → OpenSSL 自动加密 → send()
```

## 7. 学习路线建议

按依赖关系排列，建议按序深入：

| 层次 | 主题 | 核心问题 |
|------|------|---------|
| 第一层 | **对称加密 AES** | AES 是什么？怎么加密解密的？"会话密钥"怎么生成？ |
| 第二层 | **非对称加密 RSA** | 公钥和私钥的数学关系是什么？为什么公钥加密的内容公钥自己解不开？ |
| 第三层 | **哈希函数 / 数字签名** | 签名是怎么证明数据来自某人且未被篡改的？Hash 和签名的关系是什么？ |
| 第四层 | **证书体系 CA** | CA 是谁？为什么要信任 CA？root.crt 从哪来的？签名验证的具体过程？ |
| 第五层 | **TLS 协议** | TLS 握手每一步分别做了什么？TLSv1 vs TLSv1.2 vs TLSv1.3 有什么区别？ |
| 第六层 | **OpenSSL 库** | mosquitto 是怎么调用 OpenSSL 的？SSL_CTX / SSL 对象是什么？ |
| 第七层 | **OAuth2 + Base64** | Base64 为什么不是加密？OAuth2 Token 是怎么用来认证设备身份的？ |
