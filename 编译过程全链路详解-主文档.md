# 编译过程全链路详解 — 主文档

文档主线：从源码到机器码，GCC 内部做四件事（预处理→编译→汇编→链接），make 管理这四件事的执行顺序，交叉编译只是把编译器换成了 ARM 版本。这条线串起：编译原理 → 构建系统类型 → 库的依赖与传递 → 交叉编译 → 本工程编译流水线。

## 第一章：编译的本质 — GCC 内部的四件事

`gcc -o hello hello.c` 这一行命令，GCC 内部执行了四个步骤。每一步输入一种文件，输出一种文件：

```
hello.c（文本源码）
  │ 第1步：预处理 (cpp)
  ▼
hello.i（展开的源码）
  │ 第2步：编译 (cc1)
  ▼
hello.s（汇编代码）
  │ 第3步：汇编 (as)
  ▼
hello.o（目标文件，二进制机器码）
  │ 第4步：链接 (ld)
  ▼
hello（可执行文件，二进制机器码）
```

### 第一步：预处理（Preprocessor，命令 cpp）

把 `#` 开头的指令全部处理掉：

```
输入 hello.c:
  #include <stdio.h>
  #define MAX 100
  int main() { printf("%d", MAX); return 0; }

预处理后 hello.i:
  → #include <stdio.h> → 把 stdio.h 的 2000 行内容原样插入到这里
  → #define MAX 100 → 把源码中所有 MAX 替换为 100
  → #ifdef __linux__ → 条件编译，保留匹配平台的部分，删除不匹配的

输出：一个膨胀后的纯 C 源码文件（通常几万行），没有 # 号了
```

**交叉编译无关**：预处理不涉及 CPU 架构，x86 和 ARM 的 `#include` 展开逻辑完全一样。唯一的区别是条件编译——`#ifdef __arm__` 在 ARM 工具链中会保留，`#ifdef __x86_64__` 会被删除。

### 第二步：编译（Compiler，命令 cc1）

把预处理后的文本源码转成汇编代码。这是最难的一步——编译器要做词法分析、语法分析、语义分析、优化、代码生成。

```
输入 hello.i:（文本）
  int main() { int a = 5; return a + 3; }

输出 hello.s:（ARM 汇编代码）
  main:
      push {fp}              ← 保存栈帧指针
      mov  r0, #5            ← 把常量 5 装入寄存器 r0
      add  r0, r0, #3        ← r0 = r0 + 3
      pop  {fp}              ← 恢复栈帧指针
      bx   lr                ← 返回（跳转到链接寄存器中的地址）

如果没有交叉编译，默认生成 x86_64 汇编：
  main:
      mov  eax, 5            ← x86 用 eax 寄存器，ARM 用 r0
      add  eax, 3
      ret                    ← x86 用 ret 指令返回，ARM 用 bx lr
```

**为什么交叉编译在这里不同**：同样的 `return a + 3`，x86_64 和 ARM 的汇编指令完全不同。ARM 特有的寄存器命名（r0-r15）、指令格式（三操作数 `add r0, r0, #3`）、函数调用约定（参数传 r0-r3）和 x86 完全不一样。`arm-linux-gnueabihf-gcc` 的 cc1 组件生成 ARM 汇编，`gcc` 的 cc1 组件生成 x86_64 汇编。

### 第三步：汇编（Assembler，命令 as）

把汇编代码翻译成二进制机器码。这一步是机械的一对一映射。

```
输入 hello.s:
  mov r0, #5    →  输出: E3A00005（4 字节 ARM 机器码）
  add r0, r0, #3 →  输出: E2800003
  bx  lr         →  输出: E12FFF1E

输出 hello.o: ELF 格式的目标文件
  包含：
    - .text 段：上面的机器码
    - .symtab 段（符号表）：
        main: 地址 0x0000, 类型 FUNC
        printf: 地址 0x0000, 类型 UND（未定义 → 需要链接器去找）
    - .rel.text 段（重定位表）：记录哪些地址需要在链接时修正
```

符号表中 `main` 的类型是 FUNC（已定义），`printf` 的类型是 UND（未定义——这个符号在别的文件里）。链接器看到 UND 符号，就知道要去外部库中找 `printf` 的实现。

### 第四步：链接（Linker，命令 ld）

一个可执行文件通常由多个 `.o` 文件和多个外部库组成。链接器把所有东西拼起来，把 UND 符号全部解析掉。

```
输入：
  hello.o（符号：main=FUNC, printf=UND）
  /usr/lib/libc.so（包含 printf 的实现）

链接器做的事：
  1. 符号解析：hello.o 需要 printf → 去 libc.so 找到 printf 的地址
  2. 重定位：hello.o 里调用 printf 的机器码地址原来是 0x0000（占位符），
              链接器把这个地址改成 printf 在最终可执行文件中的实际地址
  3. 合并段：所有 .o 的 .text 段拼在一起，
             所有 .o 的 .data 段拼在一起，
             计算每个符号的最终地址
  4. 写 ELF 头：记录入口地址（_start，由 crt0.o 提供）

输出 hello: 完整的 ELF 可执行文件
```

**动态链接**：本工程的程序不是把所有库的代码都塞进可执行文件——`.so` 文件在运行时才加载。链接器在可执行文件的 `.dynamic` 段里记录了运行时需要加载的 `.so` 列表，运行时由动态链接器（`/lib/ld-linux-armhf.so.3`）去加载。

```
readelf -d zksort | grep NEEDED
  NEEDED  libmosquitto.so.1       ← 运行时需要这个库
  NEEDED  libQtCore.so.4
  NEEDED  libssl.so.1.0.0
  NEEDED  libcrypto.so.1.0.0
```

## 第二章：构建系统 — 谁来管理这四个步骤

### 2.1 Makefile — 最原始的构建系统

Makefile 是一个文本文件，定义**规则**：目标文件依赖哪些源文件，用什么命令生成。

```makefile
# 规则格式：
# target: dependencies
# 	command

hello: hello.o
	gcc hello.o -o hello

hello.o: hello.c
	gcc -c hello.c -o hello.o

clean:
	rm -f hello hello.o
```

执行 `make` 时：
1. make 找第一个目标（hello）
2. 发现 hello 依赖 hello.o → 检查 hello.o 是否存在且比 hello.c 新
3. 如果不存在或过期 → 执行 `gcc -c hello.c -o hello.o`
4. hello.o 生成后 → 执行 `gcc hello.o -o hello`
5. 完成

**增量编译**：如果只改了 hello.c，make 只重编 hello.o 再链接，不会重编没修改的 .o 文件。15 万行的工程如果每次全量编译要几分钟，增量编译只要几秒。

**本工程 mosquitto 用的就是这种方式**——源码下有一个 `Makefile`，执行 `make` 时读取它。

### 2.2 Configure 脚本 — OpenSSL 的构建系统

OpenSSL 的 `Configure` 是一个 Perl 脚本，执行后会生成两个文件：
- `Makefile` — 编译规则
- `configdata.pm` — 编译参数（目标平台、工具链、安装路径）

```
./Configure linux-aarch64 shared --prefix=/tmp/openssl-arm
  ↓
  1. 读取配置参数（linux-aarch64 = ARM 64 位 Linux）
  2. 探测系统环境（有没有 Perl、有没有 make、编译器版本）
  3. 生成 Makefile 和 configdata.pm
  ↓
make
  ↓
  读 Makefile → 编译
```

和 mosquitto 的 Makefile 一样，最终都是 make。区别只是 mosquitto 的 Makefile 是作者手写的，OpenSSL 的 Makefile 是 Configure 脚本生成的。

### 2.3 qmake — Qt 的构建系统

之前讲过的，读 `.pro` 文件，生成 Makefile，然后 make。

### 2.4 三种构建系统的对比

| | Makefile | Configure | qmake |
|------|---------|---------|------|
| 用什么写 | 手写文本 | Perl 脚本 | .pro 文件 |
| 怎么指定平台 | 不关心，用户自己写 CC= | linux-aarch64 参数 | QMAKE_CC 变量 |
| 怎么指定编译器 | CC= 变量 | --cross-compile-prefix | QMAKE_CC/QMAKE_CXX |
| 为什么不同 | 不同项目选择了不同工具 | — | — |

**结论：所有构建系统最终都是生成 Makefile，然后 make 执行编译。工具不同只是因为不同项目选了不同的生成方式。**

## 第三章：库的依赖与传递

### 3.1 头文件（.h）的作用——编译阶段

```c
// mosquitto 源码 tls_mosq.c 中：
#include <openssl/ssl.h>
//       ↑ 编译器需要找到这个文件

// ssl.h 中声明了 OpenSSL 的函数签名：
int SSL_connect(SSL *ssl);
// ↑ 编译器看到这行，就知道 SSL_connect 返回值是 int，参数是 SSL*
//   编译器不需要知道这个函数的具体实现——只需要知道签名，用来做类型检查
```

如果编译时找不到 `ssl.h`：
```
fatal error: openssl/ssl.h: No such file or directory
```

`-I/tmp/openssl-arm/include` 就是告诉编译器"去这个目录找头文件"。

### 3.2 库文件（.so）的作用——链接阶段

```c
// tls_mosq.c 中调用了：
SSL_connect(ssl);  // 这个函数的实现在 libssl.so 里

// 编译阶段：编译器看到 ssl.h 中 SSL_connect 的声明 → 通过类型检查 → 在 .o 符号表中标记为 UND（未定义）
// 链接阶段：链接器需要找到 SSL_connect 的实际实现
```

如果链接时找不到 `libssl.so`：
```
undefined reference to 'SSL_connect'
```

`-L/tmp/openssl-arm/lib -lssl` 就是告诉链接器"去这个目录找 libssl.so"。

### 3.3 头文件和库文件的配对

```
编译阶段：-I/tmp/openssl-arm/include
  用途：让编译器找到 openssl/ssl.h
  不涉及 .so 文件

链接阶段：-L/tmp/openssl-arm/lib -lssl -lcrypto
  用途：让链接器找到 libssl.so 和 libcrypto.so
                    解析 UND 符号
                    在 zksort 的 NEEDED 列表里写上 libssl.so.1
  不涉及 .h 文件
```

**头文件和库文件必须来自同一次编译**——如果用 ARM 版的头文件配 x86 版的 `.so`，结构定义可能不一致，链接时可能通过但运行时崩溃。

---

编译的本质、构建系统、库依赖全讲完了。交叉编译不是一套新原理——把 x86 编译器换成 ARM 编译器，四步流程完全一样。

## 第四章：交叉编译 vs 本地编译

### 4.1 对比

| | 本地编译 | 交叉编译（本工程 OpenSSL） |
|------|---------|---------|
| 编译器 | `gcc` | `aarch64-none-linux-gnu-gcc` |
| 编译器内部 cc1 | 生成 x86_64 汇编 | 生成 ARM 汇编 |
| 编译器内部 as | 生成 x86_64 机器码 | 生成 ARM 机器码 |
| 链接的库 | /usr/lib/libc.so (x86) | /tmp/openssl-arm/lib/libssl.so (ARM) |
| 运行在哪 | 这台机器 | ARM 板卡 |

### 4.2 唯一区别

编译器路径不同。四步预处理→编译→汇编→链接完全一样。Makefile 不用改——你只是传了一个不同的 `CC=` 给它。

## 第五章：本工程的编译流水线

```
1. 交叉编译 OpenSSL
   源码 → Configure(ARM) → make → libssl.so + libcrypto.so
   ↓ 产物放在 /tmp/openssl-arm/

2. 用 OpenSSL 产物交叉编译 mosquitto
   源码 → make(ARM, CFLAGS指向/tmp/openssl-arm/include,
                    LDFLAGS指向/tmp/openssl-arm/lib)
   ↓ 产物：libmosquitto.so + libmosquittopp.so
   ↓ 复制到 3rdparty/mqtt/

3. qmake + make 编译 zksort
   zksort.pro → qmake → Makefile → make(ARM)
   链接：libmosquitto.so + libQtCore.so + libssl.so + libcrypto.so...

4. 部署到 ARM 板卡
   /app/zksort
   /lib/libmosquitto.so.1, libssl.so, libcrypto.so...
   /app/root.crt
   ldconfig 刷新缓存
```

## 附录：编译错误速查

| 错误信息 | 原因 | 解决 |
|---------|------|------|
| `fatal error: openssl/ssl.h: No such file or directory` | 编译器找不到头文件 | 检查 `-I` 路径是否正确 |
| `undefined reference to 'SSL_connect'` | 链接器找不到函数实现 | 检查 `-L` 路径和 `-lssl` |
| `cannot open shared object file` | 运行时找不到 .so | 检查 `/lib/` 下有对应 .so，跑 `ldconfig` |
| `ELF file OS ABI invalid` | 目标文件架构不匹配 | 确认用的是交叉编译器而非本机编译器 |
| `illegal instruction` | 在 x86 机器上跑了 ARM 程序 | 需要交叉编译或模拟器 |
