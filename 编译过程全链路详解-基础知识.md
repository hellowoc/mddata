## 一、GCC（GNU Compiler Collection）

GCC 是一组编译器，不是一个。它包含：

| 组件 | 功能 | 对应可执行文件 |
|------|------|-------------|
| 预处理器 | 处理 #include, #define, #ifdef | cpp |
| C 编译器 | C 源码 → 汇编 | cc1 |
| C++ 编译器 | C++ 源码 → 汇编 | cc1plus |
| 汇编器 | 汇编 → 目标文件 | as（GNU Assembler） |
| 链接器 | 目标文件 + 库 → 可执行文件 | ld（GNU Linker），通常通过 gcc 间接调用 |

`gcc hello.c -o hello` 内部按顺序调了 cpp → cc1 → as → ld。

### gcc vs g++

```
gcc   → 编译 .c 文件（C 语言），不会自动链接 libstdc++
g++   → 编译 .cpp 文件（C++），自动链接 libstdc++ 和 C++ 标准库

对于 .c 文件，两者行为相同。对于 .cpp 文件，必须用 g++。
```

## 二、文件扩展名对照

| 扩展名 | 含义 | 类型 | 谁产生 |
|--------|------|------|--------|
| .c / .cpp | 源文件 | 文本 | 你写的 |
| .h / .hpp | 头文件 | 文本 | 你写的或库提供的 |
| .i / .ii | 预处理后的源码 | 文本 | cpp |
| .s | 汇编代码 | 文本 | cc1 |
| .o | 目标文件 | 二进制（ELF） | as |
| .a | 静态库 | 二进制（ELF） | ar 归档器 |
| .so | 动态库（共享库） | 二进制（ELF） | ld |
| （无扩展名） | 可执行文件 | 二进制（ELF） | ld |

## 三、静态库 vs 动态库（.a vs .so）

### 静态库 .a

```
一个 .a 文件 = 多个 .o 文件用 ar 打包在一起

创建：ar rcs libfoo.a foo.o bar.o baz.o
使用：gcc main.o -L. -lfoo -o program
      链接器把 foo.o, bar.o, baz.o 的代码全部复制到 program 里
      运行时不需要 libfoo.a，因为代码已经在 program 里了
```

### 动态库 .so

```
创建：gcc -shared -fPIC foo.o bar.o -o libfoo.so
使用：gcc main.o -L. -lfoo -o program
      链接器只在 program 的 NEEDED 列表里写上 "libfoo.so"
      运行时动态链接器去加载 libfoo.so，代码不复制到 program 里
```

### 对比

| | 静态库 .a | 动态库 .so |
|------|---------|---------|
| 文件大小 | 编译后 program 体积大 | program 体积小 |
| 运行时依赖 | 不依赖库文件 | 必须有 .so 文件 |
| 库更新 | 需重新编译 program | 替换 .so 即可，program 不用动 |
| 内存占用 | 每个进程独立一份副本 | 多个进程共享同一份 .so |
| 本工程使用 | 不用 | 所有第三方库都用 .so |

## 四、符号表（Symbol Table）

目标文件中的符号表记录了这个文件定义了哪些函数/变量、引用了哪些外部函数/变量。

```bash
nm hello.o
# 00000000 T main         ← T = 在 .text 段中定义（已实现）
# 00000000 D global_var   ← D = 在 .data 段中定义（已实现）
#          U printf       ← U = 未定义（extern，需要链接器去找）
```

链接器把所有 `.o` 的符号表汇总，把 U 符号全部解析——找不到就报 `undefined reference`。

## 五、ELF 格式（Executable and Linkable Format）

Linux 下 `.o`、`.so`、可执行文件的统一格式。

```
┌──────────────────┐
│ ELF Header       │  ← 魔数(0x7F 'E' 'L' 'F')、架构(ARM/x86)、类型(可执行/.so/.o)
├──────────────────┤
│ Program Headers  │  ← 只有可执行文件和 .so 有，告诉内核如何加载到内存
├──────────────────┤
│ .text   段       │  ← 机器码，只读
│ .rodata 段       │  ← 只读数据（字符串常量）
│ .data   段       │  ← 已初始化的全局变量
│ .bss    段       │  ← 未初始化的全局变量
├──────────────────┤
│ .symtab  段      │  ← 符号表
│ .dynamic 段      │  ← 依赖的 .so 列表和路径
│ .rel.text 段     │  ← 重定位表
├──────────────────┤
│ Section Headers  │  ← 元数据，运行时不需要
└──────────────────┘
```

查看 ELF 信息：
```bash
file zksort                    # 文件类型和架构
readelf -h zksort              # ELF 头
readelf -d zksort              # 动态链接信息（NEEDED 列表）
nm zksort                      # 符号表
ldd zksort                     # 运行时依赖的 .so 及加载路径
```

## 六、Makefile 基础

### 变量

```makefile
CC = gcc                       # 定义变量
CFLAGS = -Wall -O2 -g          # 编译选项
LDFLAGS = -L. -lfoo            # 链接选项

hello: hello.o
	$(CC) $(LDFLAGS) hello.o -o hello   # $(CC) 引用变量
```

### 自动变量

```makefile
hello.o: hello.c
	gcc -c $< -o $@
	# $< = 第一个依赖（hello.c）
	# $@ = 目标（hello.o）
	# $^ = 所有依赖
```

### 隐式规则

```makefile
# 不需要写这条规则，make 自动知道 .c → .o 的编译命令
# hello.o: hello.c
# 	gcc -c hello.c -o hello.o
```

### PHONY 目标

```makefile
.PHONY: clean     # 声明 clean 不是真实文件名
clean:
	rm -f *.o hello
# 如果没有 .PHONY，而恰好有一个名为 clean 的文件，make clean 不执行
```

## 七、-I、-L、-l 参数详解

```
gcc 命令的参数分为两类：

编译阶段参数（控制编译器行为）：
  -I/path/to/include    → 添加头文件搜索路径
  -DDEBUG               → 定义宏 #define DEBUG
  -O2                   → 优化等级 2
  -g                    → 生成调试信息
  -Wall                 → 开启所有警告
  -fPIC                 → 生成位置无关代码（编译 .so 必须加）
  -c                    → 只编译和汇编，不链接（产生 .o）

链接阶段参数（控制链接器行为）：
  -L/path/to/lib        → 添加库搜索路径
  -lfoo                 → 链接 libfoo.so（或 libfoo.a）
  -shared               → 生成共享库 .so
  -Wl,-rpath,/app/lib   → 在 ELF 中写入 RPATH
  -o output             → 输出文件名
```

**为什么 -I 和 -L 是分开的**：-I 是编译阶段用的（编译器在 .c→.o 时查头文件）。-L 是链接阶段用的（链接器在 .o→可执行文件时查库文件）。一个管编译，一个管链接，不能互换。

## 八、交叉编译工具链命名规则

```
aarch64-none-linux-gnu-gcc
│       │    │   │  │
│       │    │   │  └── gcc 编译器
│       │    │   └────── gnu 使用 GNU C 库 (glibc)
│       │    └────────── linux 目标操作系统是 Linux
│       └─────────────── none 无供应商（通用嵌入式），区别于 apple、android
└─────────────────────── aarch64 ARM 64 位架构

arm-linux-gnueabihf-gcc
│   │    │   │  │
│   │    │   │  └── hf = hard float，硬件浮点
│   │    │   └────── eabi = Embedded ABI
│   │    └────────── linux
│   └─────────────── 32 位 ARM（不带 64）
└─────────────────── arm
```

## 九、常见编译错误速查

| 错误 | 含义 | 发生在哪个阶段 |
|------|------|-------------|
| `expected ';' before '}'` | 语法错误，少写了分号 | 第二步：编译 |
| `undefined reference to 'foo'` | 链接器找不到 foo 的实现 | 第四步：链接 |
| `cannot find -lfoo` | 链接器在 -L 路径中找不到 libfoo.so | 第四步：链接 |
| `No such file or directory` | 找不到头文件（#include 的文件不存在） | 第一步：预处理 |
| `incompatible pointer type` | 指针类型不匹配，传了 int* 给期待 char* 的参数 | 第二步：编译 |
| `multiple definition of 'foo'` | 同一个符号在多个 .o 中定义了 | 第四步：链接 |

## 十、/tmp 目录

Linux 的临时文件目录。所有用户都可读写。系统重启后自动清空。编译临时产物放这里，不会污染工程目录。

```bash
# 本工程编译 OpenSSL 时：
--prefix=/tmp/openssl-arm
# 意思是"安装到 /tmp/openssl-arm"，编译完成后手动把 .so 复制到工程。
# 用 /tmp 因为不是要安装到系统——只要编译产物，不要污染系统路径。
```
