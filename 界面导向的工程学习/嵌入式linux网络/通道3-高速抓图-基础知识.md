# 通道3 高速抓图 — 基础知识清单

按在 [[通道3-高速抓图-主文档|主文档]] 中首次出现的顺序排列。

---

## 第 1 章出现的

### 行连续性测试（§1）

IPC 相机以一帧帧的方式发送图像数据。每一行图像被拆分到多个 UDP 包中，每个包的包头包含**行号**。上位机接收时逐个检查行号是否递增——如果行号跳过了某个值（如上一包行号 100，下一包行号 103），说明丢了 2 行。

这是一个**网络质量诊断工具**，用于排查 IPC 的网络传输是否稳定。

### 全帧抓图（§1）

接收完整一帧图像（通常 400~832 行），将所有 UDP 包中的像素数据拼成完整的 BMP 位图保存到磁盘。

用途：工程师检查 IPC 相机拍摄的图像质量（清晰度、曝光、颜色）。

## 第 3 章出现的

### 图像序号 imageSeq（§3）

32 位整数（4 字节），存储在包头偏移 2~5 位置。标识这个包属于哪一帧图像。

全帧抓图模式下，一帧图像被拆成很多 UDP 包，每个包都携带相同的 `imageSeq`。上位机根据 `imageSeq` 将包归类到不同的帧中。

### 行号（§3）

16 位整数（2 字节），存储在包头偏移 6~7 位置。标识这个包属于图像的第几行。

行连续性测试的核心就是检查相邻包的**行号是否连续**。

### 分包序号（§3）

一行 1024 像素 × 3 通道(RGB) = 3072 字节数据。但一个 UDP 包的有效载荷通常在 1024 字节左右（受以太网 MTU 1500 的限制）。所以一行需要拆成 3 个 UDP 包发送，分包序号标识这是本行的第几个分片。

### MTU（§3）

MTU = Maximum Transmission Unit（最大传输单元）。以太网的 MTU 是 **1500 字节**。

减去 IP 头（20 字节）和 UDP 头（8 字节），一个 UDP 包最多携带 **1472 字节**的用户数据。如果一行图像数据超过 1472 字节，就必须分片。

### m_packetPerLine（§3）

`MyFastNetWorkThread` 的成员变量。每行需要多少个 UDP 包：

```cpp
m_packetPerLine = (pixels × 3) / bytesPerPacket;
// 1024像素 × 3通道 / 1024字节每包 ≈ 3 包/行
// 2048像素 × 3通道 / 1024字节每包 ≈ 6 包/行
```

## 第 4 章出现的

### udpSocketInit()（§4-Step3）

构造函数中调用。初始化 BSD socket 并绑定。这个函数**不启动线程**——线程在外部调用 `start()` 时才启动。

这个设计意味着：程序启动时 socket 就绑定好了（可以开始接收数据），但线程还没跑。因为 IPC 不一定一直在推送图像——只在用户触发测试时才启动线程去收。

### socket()（§4-Step3）

见 [[references/Socket编程基础#socket() — 创建 socket]]。

### setsockopt SO_RCVBUF（§4-Step3）

与通道1 一样的设置——8MB 接收缓冲区。因为 IPC 图像数据量也非常大。

### udpServerInit()（§4-Step4）

遍历配置中所有启用的 IPC 相机，为每台建立独立的 `addr_remote[view][unit]`。

**二维数组寻址**：`addr_remote[j][k]` 中 `j` = 视野号（view），`k` = 单元号（unit）。一个视野可以有多个单元（如同一视野下有多台 IPC 相机），每台 IPC 的 IP 不同。

### nIpAddr[4]（§4-Step4）

`struCnfg.struIpcCameraParams[layer][view][unit].nIpAddr[]` 是一个 `int` 数组，存储 IPC 相机的 4 段 IP 地址：

```cpp
nIpAddr[0] = 192
nIpAddr[1] = 168
nIpAddr[2] = 1
nIpAddr[3] = 10    // → 完整 IP = 192.168.1.10
```

只取了 `[4]` 看起来不对——应该是 4 个元素（索引 0~3），`nIpAddr[4]` 取的是第 5 个元素。这可能是：
1. 数组实际定义更大（`nIpAddr[6]` 或类似），`[4]` 存的是第 4 段的**偏移量**而非直接值
2. 或者是一个编码——`[4]` 可能是独立的标识位而非 IP 段

从代码 `QString("192.168.1.%1").arg(nIpAddr[4])` 来看，前面三段是硬编码的，只从配置中获取最后一段。

### IPC_CAMERA_TYPE_NULL（§4-Step4）

宏（`constant.h`），值 = `0`。表示"该位置没有安装 IPC 相机"。

如果 `ipcType == IPC_CAMERA_TYPE_NULL`，跳过该位置的 IP 配置。

## 第 5 章出现的

### 对齐（§5-模式1）

当收到非法包（包头不是 `A5 5A`）时，不能简单丢弃整个缓冲区，而是要找到下一个 `A5 5A` 的位置，从那里重新开始解析。这叫**字节对齐**或**帧同步**：

```
收到数据: XX YY A5 5A XX YY A5 5A XX ...
          └─非法─┘└────合法包头────┘

丢弃前 2 个字节，从 A5 5A 开始解析。
```

### 图像拼接（§5-模式1）

`m_packetPerLine > 1` 时，一行的像素数据分布在多个 UDP 包中。需要按分包序号拼接成完整的一行。

### imageCombineBuf（§5-模式2）

`MyFastNetWorkThread` 的成员数组，大缓冲区，用于将收到的 UDP 包拼接成完整帧。

### saveBMPFile()（§5-模式2）

`MyFastNetWorkThread` 的方法（`myfastnetwork.cpp`）。将原始像素数据封装为 BMP 文件格式（写文件头 BITMAPFILEHEADER + 信息头 BITMAPINFOHEADER + 像素数据），保存到 `/sdcard/bmp/`。

## 第 6 章出现的

### FastNetObj（§6）

`myudpthread.h:152-170` 中定义的类。与 `BackgroudObj` 姐妹类，但绑定在 **eth1**（`192.168.1.100:20000`）而非 eth0。

用于向 IPC 相机发送**高速控制命令**（如启动/停止抓图），与接收图像数据的 `MyFastNetWorkThread` 配合使用。

### 命令发送 vs 数据接收（§6）

通道3 的收发是**分离的**：
- 发命令：`FastNetObj`（属于通道2 的 `SendThread`，用 Qt QUdpSocket，绑定 192.168.1.100:20000）
- 收数据：`MyFastNetWorkThread`（独立线程，用 BSD socket，绑定 192.168.1.100:20001）

为什么分离？因为命令是低频的（偶尔发一次"开始/停止"），图像是高频的（持续高速推送）。用不同的 socket 和线程避免干扰。

## 第 7 章出现的

### 硬编码 IP vs 配置读取（§7）

通道1 的远端 IP 是硬编码的 `162.254.129.10`（只有一块 FPGA 板，地址固定）。通道3 的远端 IP 是从配置读取的（`192.168.1.x`），因为可能有**多台 IPC 工控机**，每台 IP 不同。

### 线程启动时机（§7）

通道1 的线程在用户点击采图按钮时启动（`udpStart()` → `start()`）。通道3 的线程也在用户点击测试按钮时启动。但通道3 的 socket 初始化在**构造函数**中就完成了（可以提前绑定端口），这是为了确保端口不被其他程序占用。
