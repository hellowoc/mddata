## 学习目标

从零基础理解嵌入式 Linux 网络通信，到完全掌握色选机工程中每个网络相关界面控件的工作原理和底层代码实现。

## 第一阶段：认识你的网络拓扑（1天）

### 1.1 先理解物理连接

你的色选机 Linux 工控机上有 **两张网卡**，分别接了两个独立的物理网络：

```
┌─────────────────────────────────────────────────────┐
│                  你的 Linux 工控机                    │
│                                                     │
│  eth0: 162.254.129.100  ──── FPGA 相机网段           │
│  eth1: 192.168.1.100    ──── IPC AI 工控机网段        │
└─────────────────────────────────────────────────────┘
```

- **eth0 (162.254.129.x)**：连接 FPGA 相机板，传输波形数据、图片、喷阀计数
- **eth1 (192.168.1.x)**：连接 IPC AI 工控机（深度学习推理），传输 AI 检测结果、高速抓图

### 1.2 要搞清楚的问题
- 什么是以太网？IP 地址、子网掩码是什么？
- 什么是 UDP？为什么这里全用 UDP 而不是 TCP？
- 什么是端口？为什么一个 IP 上可以绑定多个端口？

### 1.3 对应代码文件
- [mynetwork.h/cpp](zksort/bus/mynetwork.h) — FPGA 相机 UDP 通信
- [myfastnetwork.h/cpp](zksort/bus/myfastnetwork.h) — IPC 高速抓图 UDP 通信
- [myudpthread.h/cpp](zksort/bus/myudpthread.h) — IPC 命令 UDP 通信

## 第二阶段：理解 UDP 通信的基本套路（2天）

### 2.1 三种 UDP 通信模式对比

| 通信通道 | 本地绑定 | 发送目标 | 用途 |
|---------|---------|---------|------|
| FPGA 相机 | 162.254.129.100:**8081** | 162.254.129.10 | 波形采集、图像抓取 |
| IPC 命令 | 162.254.129.100:**8081** (收 **8082**) | 162.254.129.10:**5001** | AI 参数配置、状态查询 |
| IPC 高速抓图 | 192.168.1.100:**20001** (收 **20000**) | 192.168.1.x:**20000** | 图像行连续性测试、全帧抓图 |

### 2.2 代码阅读顺序

**Step 1**: [myudpthread.cpp](zksort/bus/myudpthread.cpp) — 最核心的 IPC 命令系统
- `BackgroudObj` 类：如何 `bind()` 端口，如何 `readPendingDatagrams()` 收包
- `udp_header` 结构体：12 字节自定义二进制协议头
- `processTheDatagram()`：如何根据命令字分发处理

**Step 2**: [mynetwork.cpp](zksort/bus/mynetwork.cpp) — FPGA 相机 UDP
- `MyNetWorkThread::run()`：注意它用的是 BSD socket（`socket()`/`bind()`/`select()`/`recvfrom()`），不是 Qt 的 QUdpSocket
- 包格式：`0xFF 0x00 0xA5 0x5A 0xC3 0x3C` 6 字节包头

**Step 3**: [myfastnetwork.cpp](zksort/bus/myfastnetwork.cpp) — IPC 高速抓图
- `MyFastNetWorkThread::run()`：两种模式（行连续性测试 / 全帧抓图）
- 包格式：`0xA5 0x5A` 2 字节包头 + 14 字节头 + 像素数据

### 2.3 要搞清楚的问题
- `bind()` 是干什么的？为什么收发用不同端口？
- `select()` 和 Qt 事件循环的区别？
- 自定义二进制协议为什么要 CRC 校验？

## 第三阶段：线程模型（2天）

### 3.1 工程中的线程全景

```
主线程 (GUI 事件循环)
├── mySerialSendThread1/2    (COM1/COM2 串口发送线程)
├── mySerialRecvThread1/2    (COM1/COM2 串口接收线程)
├── MyNetWorkThread          (FPGA 相机 UDP 线程)
├── MyFastNetWorkThread      (IPC 高速抓图 UDP 线程)
├── SendThread               (IPC 命令 UDP 发送线程) ── 内含 BackgroudObj + FastNetObj
├── mqttThread               (MQTT 云通信线程，仅 Linux)
├── mqttMsgParaseThread      (MQTT 消息解析线程，仅 Linux)
├── myUpdateStatusThread     (状态轮询线程：时钟/清灰/报警)
├── IpcLoadThread[]          (IPC 模型加载线程，每个相机一个)
└── ipcTestNetThread         (IPC 网络测试线程)
```

### 3.2 线程间的数据流

```
UI 控件点击
  → 信号/槽 → 主线程函数
    → pushUdpCmdData() / pushCom1CmdData() 放入队列
      → SendThread / mySerialSendThread 从队列取包发送
        → 硬件收到 → 回包
          → BackgroudObj / mySerialRecvThread 收到回包
            → processTheDatagram() 解析 → 更新全局数据结构
              → 发射信号 → UI 控件刷新
```

### 3.3 要搞清楚的问题
- QThread 的两种用法：`run()` 死循环 vs `exec()` 事件循环
- `QMutex` 和线程安全队列（[myqueue.cpp](zksort/bus/myqueue.cpp)）
- 为什么通信线程要和 GUI 线程分离？

### 3.4 对应代码文件
- [myqueue.h/cpp](zksort/bus/myqueue.h) — 环形缓冲队列

## 第六阶段：理解协议和命令字（2天）

### 6.1 IPC 通信协议（核心）

协议头定义在 [myudpthread.h](zksort/bus/myudpthread.h) 的 `udp_header`：

```
Byte 0-1:  0xA5 0x5A  (帧头标识)
Byte 2-3:  版本号
Byte 4-5:  板地址 (高字节=view, 低字节=unit)
Byte 6-7:  数据长度
Byte 8-9:  序列号 (用于请求-应答匹配)
Byte 10-11: 命令字
Byte 12-13: CRC 校验
```

### 6.2 关键命令字一览

| 命令字 | 宏名 | 功能 |
|-------|------|------|
| 0x0001 | `CMD_UDP_IPC_REQ_INFO` | 查询 IPC 信息（心跳） |
| 0x0002 | `CMD_UDP_IPC_REQ_MODEL_ABLE` | 查询可用模型列表 |
| 0x0003 | `CMD_UDP_IPC_REQ_MODEL_INFO` | 查询模型详情 |
| 0x0004 | `CMD_UDP_IPC_REQ_MODEL_LOAD` | 加载模型 |
| 0x0008 | `CMD_UDP_IPC_REQ_START_SORT` | 启动物料分选 |
| 0x0009 | `CMD_UDP_IPC_REQ_SHUTDOWN` | 关闭 IPC |
| 0x000A | `CMD_UDP_IPC_REQ_BAD_MATERIAL_COUNTS` | 查询坏料计数 |
| 0x00C0 | `CMD_CAMERA_AI_MISC_STATUS` | IPC 心跳 + 状态综合 |
| 0x00C1 | `CMD_SCREEN_GET_FILE` | 从 IPC 下载文件 |
| 0x5006 | `CMD_UDP_HT_GET_FAST_CAPTURE` | 启动高速抓图 |
| 0x5101 | `CMD_UDP_HT_GET_CAM_IMAGE_INFO_ANS` | DDR 图像数量应答 |

### 6.3 要搞清楚的问题
- 请求-应答模式：序列号如何匹配？
- 超时重传机制：`udp_packet` 中的 `sendCount` 和 `timeSend`
- 为什么 UDP 还要做 CRC？（因为自定义协议没有 TCP 的校验机制）

## 动手实践建议

1. **在 Linux 上跑命令看网络状态**：
   ```bash
   ifconfig           # 看 eth0/eth1 的 IP
   netstat -uanp      # 看哪些进程在监听 UDP 端口
   tcpdump -i eth0    # 抓包看实际通信内容
   ```

2. **加调试日志**：在 `processTheDatagram()` 里加 `qDebug()` 打印收到的命令字

3. **对照界面看代码**：打开一个设置页 → 点一个按钮 → 跟踪代码从 UI 信号 → 命令发送 → 回包处理 → UI 更新 的完整链路

4. **画时序图**：选一个功能（如"加载模型"），画出 UI / globalflow / g_Udp / IPC 之间的消息序列
