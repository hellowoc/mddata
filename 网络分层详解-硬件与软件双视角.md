# 网络分层详解 — 硬件与软件双视角

## 先给结论

分层就是把"把数据从A机器搬到B机器"这件复杂的事，拆成几层各自负责。

每层只做自己那一层的事，用下层提供的服务，给上层提供服务。这样你写 MQTT 代码的时候不用管 IP 包怎么路由，写 IP 路由的时候不用管网线里的电信号怎么编码。

---

## 一、硬件视角："数据从A到B，经过了哪些物理设备"

```
设备A（工控机）                                          设备B（云服务器）
┌─────────────────────────┐                  ┌─────────────────────────┐
│                         │                  │                         │
│  应用层  ← 你的程序      │                  │  应用层  ← 云端服务程序   │
│    zksort可执行文件      │                  │    Java/Go 服务          │
│                         │                  │                         │
│  传输层 ← Linux内核      │                  │  传输层 ← Linux内核      │
│    TCP/UDP协议栈         │                  │    TCP/UDP协议栈         │
│                         │                  │                         │
│  网络层 ← Linux内核      │    ┌────────┐   │  网络层 ← Linux内核      │
│    IP路由表              │    │ 路由器  │   │    IP路由表              │
│                         │    │        │   │                         │
│  链路层 ← 网卡驱动+硬件  │    │ 查路由表│   │  链路层 ← 网卡驱动+硬件  │
│    MAC地址               │    │ 转发IP包│   │    MAC地址               │
│                         │    └────────┘   │                         │
│  物理层 ← 网口+网线      │       │         │  物理层 ← 网口+网线      │
│    电信号/光信号          │       │         │    电信号/光信号          │
└─────────────────────────┘       │         └─────────────────────────┘
                                  │
                          中间设备只到网络层
                          （路由器不需要理解TCP，
                            更不需要理解MQTT）
```

### 关键认知

- **应用层**：只有通信的两端才处理（你的程序和云端服务程序）
- **传输层（TCP/UDP）**：也只有两端才处理。中间的设备（路由器）不碰这一层
- **网络层（IP）**：两端 + 中间路由器都处理。路由器就是靠这一层决定"下一跳往哪走"
- **链路层+物理层**：每一段物理链路都处理（设备→路由器、路由器→路由器、路由器→服务器）

### 硬件对应

| 层 | 在哪些硬件上运行 |
|------|---------|
| 应用层 | 工控机的 ARM CPU、云服务器的 x86 CPU |
| 传输层 | 同上——内核代码在 CPU 上执行 |
| 网络层 | 两端的 CPU + 路由器的 ASIC/CPU |
| 链路层 | 网卡芯片（如 Realtek RTL8211E）+ 网卡驱动 |
| 物理层 | 网口变压器 + RJ45 接口 + 网线铜芯/光纤 |

## 二、软件视角："数据从你的程序出发，经过哪些代码"

以本工程 MQTT 发一条消息为例：

```
你的代码层：
  myMqttThread->test->publish(NULL, "MqttServerClient", str, QoS=2)
  └→ mqttsrv（你的C++代码）
      └→ mosquittopp（C++ wrapper）
          └→ libmosquitto（C库）
              │  组装 MQTT PUBLISH 报文：
              │  固定头(0x3D) + 剩余长度 + 主题名 + PacketId + Payload
              │
══════════════ ══ 系统调用边界 ══ ══════════════════════════════
              │  write(fd, mqtt_packet, len)
              ▼
Linux 内核层：
  传输层(TCP)：
    tcp_sendmsg()  ← 内核函数
    │ 把 MQTT 报文 copy 到 TCP 发送缓冲区
    │ 把数据按 MSS（最大段大小，通常 1460 字节）切成 TCP 段
    │ 每个段加上 TCP 头：
    │   源端口: 随机(如 45678)
    │   目标端口: 1884
    │   序号: snd_nxt（如 1000）
    │   ACK号: rcv_nxt
    │   标志位: PSH+ACK
    │ 放到重传队列（等 ACK，超时就重发）
    │
  网络层(IP)：
    ip_queue_xmit()  ← 内核函数
    │ 每个 TCP 段加上 IP 头：
    │   源 IP: 工控机 eth0 的 IP（如 162.254.129.100）
    │   目标 IP: cloud.chinaamd.cn 解析出的 IP
    │   TTL: 64（每经过一个路由器减1，到0丢弃）
    │   Protocol: 6（=TCP）
    │ 查路由表："目标 IP 不在本地网段，走默认网关 eth0 → 路由器"
    │
  链路层：
    │ 通过 ARP 获取"下一跳"的 MAC 地址
    │ 加上以太网帧头：
    │   源 MAC: 工控机 eth0 的 MAC（如 00:0C:29:AB:CD:EF）
    │   目标 MAC: 路由器的 MAC
    │   类型: 0x0800（=IPv4）
    │
  物理层：
    网卡驱动把帧写入网卡芯片的发送 FIFO
    │
═══════════ ══ 硬件边界 ══ ════════════════════════════════
    │
    网卡芯片把数字信号 → 电平变化 → 网口变压器耦合 → RJ45 → 网线铜芯
    │ 差分信号 ±2.5V，曼彻斯特编码或 4D-PAM5
    │
    ────→ 网线 ──→ 路由器 ──→ ...（多跳）... ──→ 云服务器
```

### 从下往上收时，顺序反过来

```
网线电信号 → 网卡芯片解码 → 驱动收帧 → 链路层剥以太网头
→ IP 层剥 IP 头 → TCP 层重组 TCP 段、检查序号、发 ACK
→ data 放入接收缓冲区 → read() 系统调用
→ libmosquitto 解析 MQTT 报文 → on_message() 回调 → 你的代码
```

## 三、每层的"信封" — 封装与解封装

这是分层最核心的概念：**每一层在前一层的数据前面加上自己的头（header），就像信封套信封**。

```
MQTT PUBLISH 报文（应用层）:
┌─────────────────────────────────────────────┐
│ MQTT Header (固定头+可变头) │ Payload (JSON)   │
└─────────────────────────────────────────────┘
                    │
                    │ 进入 TCP 层，整个 MQTT 报文变成 TCP 的 "Payload"
                    ▼
┌──────────────────┬──────────────────────────┐
│ TCP Header(20B)  │ MQTT 报文（上面那整块）     │
│ 端口、序号、ACK号  │                          │
└──────────────────┴──────────────────────────┘
                    │
                    │ 进入 IP 层，整个 TCP 段变成 IP 的 "Payload"
                    ▼
┌────────────┬───────────────────────────────┐
│ IP Hdr(20B)│ TCP段（上面那整块）              │
│ 源/目标IP   │                               │
└────────────┴───────────────────────────────┘
                    │
                    │ 进入以太网层
                    ▼
┌──────────────┬──────┬───────────────────────┬──────┐
│ 以太网头(14B) │IP头  │ TCP头 │ MQTT头 │ JSON  │ FCS(4B)│
│ MAC地址/类型  │      │       │        │       │ CRC校验 │
└──────────────┴──────┴───────┴────────┴───────┴──────┘
```

**接收端解封装**：以太网层剥 MAC 头 → IP 层剥 IP 头 → TCP 层剥 TCP 头 → MQTT 报文原封不动交给应用层。

数据结构上就是**每层加自己的头，形成一个嵌套结构体**。

## 四、每层"做决策"的依据是什么

| 层 | 决策问题 | 依据的数据 | 例子 |
|------|---------|---------|------|
| 应用层 | 发什么内容 | MQTT Topic、JSON 结构 | `publish("MqttServerClient", json)` |
| 传输层(TCP) | 什么时候重发、发多快 | 序号、ACK号、窗口大小 | `tcp_sock.snd_nxt`、收到重复 ACK 触发重传 |
| 网络层(IP) | 下一跳往哪走 | 路由表 | `route -n` 查看：`default via 192.168.1.1 dev eth0` |
| 链路层 | 这段链路上发给谁 | ARP 表（IP→MAC 映射） | `arp -a` 查看：`192.168.1.1 → 00:11:22:33:44:55` |
| 物理层 | 用什么电压/频率表示 0 和 1 | 物理层协议规范 | 1000BASE-T：4 对双绞线，每对 250Mbps，PAM-5 编码 |

## 五、对应到本工程的代码

| 层 | 你直接交互的代码 | 你间接依赖的 |
|------|---------|---------|
| 应用层 | `mqttsrv.cpp`、`myhttpfileclient.cpp`、MQTT 业务逻辑 | mosquitto 库、jsoncpp |
| 传输层 | `socket()` 的参数选择（`SOCK_DGRAM`/`SOCK_STREAM`）、`setsockopt()` 调缓冲区大小 | Linux 内核 TCP/UDP 协议栈 |
| 网络层 | `inet_addr("162.254.129.100")`、`bind()` | 内核路由表、`/proc/sys/net/` |
| 链路层 | 不直接交互 | 网卡驱动（内核模块） |
| 物理层 | `system("ifconfig eth0 up/down")` 开关网卡 | 网卡硬件、FPGA 相机板 |

**你在代码里调 `socket()`、`bind()`、`connect()`，本质上是请求 Linux 内核在传输层和网络层创建并维护相应的数据结构。** 内核会为你分配 `tcp_sock` 结构体、管理序号/ACK/窗口/重传——这些你一行代码都不用写。

## 六、一句话总结每层

| 层 | 一句话 |
|------|------|
| 应用层 | **"我要发什么内容"** — 定义消息格式和业务逻辑 |
| 传输层 | **"怎么保证数据可靠到达"** — TCP 用序号+ACK+重传，UDP 什么都不保证 |
| 网络层 | **"数据从哪条路走到目标"** — IP 用路由表决定下一跳 |
| 链路层 | **"这一段物理链路上怎么传输"** — 以太网用 MAC 地址寻址 |
| 物理层 | **"0 和 1 在物理介质上怎么表示"** — 电平、光脉冲、无线电波形 |
