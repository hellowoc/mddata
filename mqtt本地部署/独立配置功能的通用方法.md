# 独立配置功能的通用方法

## 文档主线

从"拿到一个已被写好的功能代码，但不知道它在当前工程中是否可用"出发，经过四层诊断找到全部阻断点，按依赖顺序逐一修复，最终验证功能生效。全程以本工程 MQTT 配置为真实案例对照。

---

# 第一章：诊断体系

一个功能被禁用，阻断必然发生在以下四层之一或多处。每一层诊断独立进行，但修复必须按依赖顺序。

---

## 第一层：源码关卡 — 编译器能不能看到这些代码

### 1.1 模式A：#if 0 完全禁断

**现象**：整个类的定义被 `#if 0 ... #endif` 包裹，编译器根本不知道这个类存在。

```cpp
// mqttsrv.h 第77行（MQTT真实案例）
#if 0                          // ← 读到这，编译器直接跳到对应的 #endif
#ifdef Q_OS_UNIX
class mqttsrv:public mosqpp::mosquittopp { ... };
class mqttThread : public QThread { ... };
class mqttMsgParaseThread : public QThread { ... };
extern mqttThread *myMqttThread;
extern mqttMsgParaseThread *myMqttMsgParaseThread;
#endif
#endif                         // ← 跳到这，中间全部跳过
```

**排查命令**：
```bash
grep -rn '^#if 0$' zksort/ --include='*.h' --include='*.cpp' | grep -v build/
```

**输出解读**：列出所有 `#if 0` 块的位置（文件:行号）。每个命中点代表一个被禁用的代码块。

**判断是否属于目标功能**：打开文件，看 `#if 0` 后面几行的内容。MQTT 案例中，`mqttsrv.h:77` 的 `#if 0` 后面紧跟着 `class mqttsrv`，确认属于 MQTT 功能。

**修复方法 — 配对删除**：
```c
// #if 0 嵌套结构：
#if 0              // ← 深度0（删这行）
#ifdef Q_OS_UNIX   // ← 深度1
...代码...
#endif             // ← 深度0（关 Q_OS_UNIX）
#endif             // ← 深度-1（关 #if 0。删这行！）
```
从 `#if 0` 往下数。每遇到 `#if`/`#ifdef`/`#ifndef`，深度+1。每遇到 `#endif`，深度-1。深度回到 -1 时的那个 `#endif` 就是配对标。

**MQTT 案例**：`mqttsrv.h` 删了第77行的 `#if 0` 和第298行的配对 `#endif`。`mqttsrv.cpp` 删了第4行的 `#if 0` 和第4991行的配对 `#endif`。

**常见错误**：删 `#if 0` 但忘记删配对的 `#endif` → 编译报 `#endif without #if`。

### 1.2 模式B：#ifdef 平台分割

**现象**：代码只在特定平台下编译，通过 `#ifdef` 条件控制。

```cpp
// globalflow.cpp 第10844行（MQTT真实案例）
#ifdef Q_OS_UNIX
    myHttpFileClient = new MyHttpFileClient;
    myMqttThread = new mqttThread;
    myMqttMsgParaseThread = new mqttMsgParaseThread;
#endif
```

**排查命令**：
```bash
grep -rn '#ifdef Q_OS_UNIX\|#ifdef Q_OS_WIN' zksort/ --include='*.cpp' | grep -v build/
```

**判断标准**：当前系统是 Linux，`Q_OS_UNIX` 在编译时被定义（Qt 自动定义），所以 `#ifdef Q_OS_UNIX` 块内的代码生效。如果功能只在 `#ifdef Q_OS_WIN` 里，那在当前 Linux 环境就不存在。

**修复**：一般不需要改——这是设计如此。除非你在 x86_64 Linux 上开发但功能被错误地放在了 `#ifdef Q_OS_WIN` 块里。

### 1.3 模式C：逐行注释

**现象**：代码没有被 `#if 0` 包裹，但关键调用行前面有 `//`。

```cpp
// globalflow.cpp 第10846-10850行（MQTT真实案例，修复前）
#ifdef Q_OS_UNIX
    myHttpFileClient = new MyHttpFileClient;
//    myMqttThread = new mqttThread;           // ← 这行被注释
//    myMqttMsgParaseThread = new mqttMsgParaseThread;  // ← 这行被注释
    if (ipForInterface == "162.254.129.100"){
//        myMqttThread->start();               // ← 这行被注释
//        myMqttMsgParaseThread->start();       // ← 这行被注释
    }
#endif
```

**排查命令**：
```bash
grep -rn '//.*new\|//.*start()\|//.*Thread\|//.*Srv' zksort/global/ --include='*.cpp'
```

**关键区别**：`#if 0` 禁的是类的"声明"（编译器看不到这个类），`//` 禁的是类的"调用"（类存在但没有被创建）。两者可能同时存在（MQTT 就是这样），必须都修。

**修复**：直接删除 `//`，恢复原代码。

### 1.4 模式D：头文件缺失

**现象**：`#include` 的文件不存在，或者路径不在 `INCLUDEPATH` 范围内。

```bash
# 查 INCLUDEPATH 是否包含所需头文件的目录
grep -A30 'INCLUDEPATH' zksort.pro
```

**MQTT 案例**：`bus/mosquitto.h` 和 `bus/mosquittopp.h` 存在于 `zksort/bus/` 目录，而 `zksort.pro` 的 INCLUDEPATH 包含 `$$PWD/bus`，所以能找到。

### 1.5 源码关卡小结

| 模式 | 命令 | 关键坑 |
|------|------|------|
| `#if 0` | `grep -rn '^#if 0' zksort/` | 删除时必须配对删 `#endif` |
| `#ifdef` | `grep -rn '#ifdef Q_OS_UNIX'` | 确认当前平台是否匹配 |
| `//` 注释 | `grep -rn '//.*new\|//.*start()'` | 和 #if 0 可能同时存在 |
| 头文件 | `grep INCLUDEPATH zksort.pro` | .h 在不在这些目录里 |

## 第二层：链接关卡 — 库文件是否存在且可用

### 2.1 LIBS 被注释

**现象**：源码编译通过了，但链接时找不到库的符号。

```qmake
# zksort.pro 第589行（MQTT真实案例，修复前）
#LIBS += -L$$PWD/../3rdparty/mqtt -lmosquittopp -lmosquitto
```

**排查命令**：
```bash
grep -n '#.*-l\|#LIBS' zksort.pro
```

**解读**：行首有 `#` = 被注释，链接器不会处理这一行。

**命令语法**：
- `-L/path/to/libs`：链接器去这个目录找 `.so` 文件。可以有多个 `-L`，按顺序搜索
- `-lmosquitto`：链接 `libmosquitto.so`。链接器自动加 `lib` 前缀和 `.so` 后缀
- `-lssl -lcrypto`：链接 OpenSSL。多个库用空格分隔

### 2.2 3rdparty 目录为空

**现象**：`.pro` 的 LIBS 指定了 `-L路径`，但那个目录下没有任何 `.so` 文件。

```bash
# 批量检查所有 3rdparty 目录
for d in 3rdparty/*/; do
    name=$(basename "$d")
    count=$(find "$d" -name '*.so*' -o -name '*.a' 2>/dev/null | wc -l)
    echo "$name: $count 个库文件"
done
```

**MQTT 案例输出**：
```
mqtt: 0 个库文件    ← 空的！需要部署
svm: 0 个库文件     ← 同样未启用
qrencode: 0 个库文件
fToFp: 0 个库文件
```

### 2.3 .so 存在但架构不匹配

**现象**：编译时有三方库，但是 ARM 架构的，当前 x86_64 系统无法链接。

```bash
file 3rdparty/*/lib/unix/*.so.1 2>/dev/null
```

**输出解读**：
```
3rdparty/log4qt/lib/unix/liblog4qt.so.1.0.0: ELF 64-bit LSB shared object, x86-64
3rdparty/jsoncpp/lib/unix/libjsoncpp.so.1.0.0: ELF 64-bit LSB shared object, x86-64
```

`x86-64` = 当前系统可用。如果显示 `ARM aarch64` = 需要重新编译 x86_64 版本。

### 2.4 系统库版本与工程头文件不兼容

**MQTT 的真实坑**：系统 `libmosquitto-dev` 是 2.0.11，但工程 `bus/mosquitto.h` 是 1.x 版本的头文件（看版权年份 2010-2014）。2.x 删除了 C++ wrapper（`mosquittopp` 类），与工程不兼容。

**诊断方法**：
```bash
# 看系统装的版本
dpkg -l | grep 库名-dev

# 看工程用的头文件版本
head -5 bus/mosquitto.h

# 看系统头文件的 API 是否有工程所需的类
grep -r '需要的类名或函数名' /usr/include/
```

**修复**：不能 `apt install` 最新版——要下载/编译与头文件版本匹配的库源码。

```bash
# MQTT 案例：编译 mosquitto 1.6.15（最后一个包含 mosquittopp 的版本）
curl -L -o mosquitto-1.6.15.tar.gz https://mosquitto.org/files/source/mosquitto-1.6.15.tar.gz
tar xzf mosquitto-1.6.15.tar.gz && cd mosquitto-1.6.15
make WITH_TLS=yes -j$(nproc)
# 把编译产物复制到 3rdparty/mqtt/
cp lib/libmosquitto.so.1 ../../3rdparty/mqtt/
cp lib/cpp/libmosquittopp.so.1 ../../3rdparty/mqtt/
cd ../../3rdparty/mqtt
ln -sf libmosquitto.so.1 libmosquitto.so
ln -sf libmosquittopp.so.1 libmosquittopp.so
```

### 2.5 SOURCES vs LIBS 不同步陷阱

```qmake
unix:{
    SOURCES +=  bus/mqttsrv.cpp    # 源码会被编译
    #LIBS += -lmosquittopp          # 但库被注释了！
}
```

源码在 SOURCES 里但库没链接 → 如果源码没被 `#if 0` 保护，链接时报 `undefined reference`。MQTT 能编译通过是因为源码同时被 `#if 0` 保护了——这两个机制被同步禁用了。**修的时候必须两个同时打开。**

## 第三层：初始化关卡 — main 函数会不会走到这段代码

### 3.1 定位初始化入口

整个程序的初始化在 `globalflow.cpp` 的 `initAll()` 函数中。所有线程、服务对象都在这里创建。

```bash
grep -n 'void GlobalFlow::initAll()' zksort/global/globalflow.cpp
```

### 3.2 三种静默失败

**形式1：创建代码被注释**
```cpp
// myMqttThread = new mqttThread;    // 对象根本不存在
```

**形式2：启动代码被注释**
```cpp
myMqttThread = new mqttThread;      // 对象创建了
// myMqttThread->start();           // 但没有 start()
```

**形式3：条件判断导致跳过**（最隐蔽）
```cpp
myMqttThread = new mqttThread;
if (ipForInterface == "162.254.129.100"){  // ← 条件不满足
    myMqttThread->start();                  // 永远不会走到
}
```

形式3 的代码看起来"正常"——没有 `//` 也没有 `#if 0`，但因为配置文件里字段为空或 `if` 条件不满足，功能静默不运行。

### 3.3 排查方法

```bash
# 搜目标功能相关类的创建和启动
grep -n 'new.*Thread\|new.*Srv\|->start()' zksort/global/globalflow.cpp
```

对每一行检查：是否被 `//` 注释？是否在 `#if` 条件里？是否依赖于某个外部字段的值？

## 第四层：运行时关卡 — 程序跑起来后能不能正常工作

### 4.1 配置文件缺失

程序从写死路径读配置。从源码逆推所需文件：

```bash
grep -rn '/sdcard/\|/app/\|./lang/' zksort/ --include='*.cpp' | grep -v build/
```

**MQTT 逆推结果**：
```cpp
// globalflow.cpp
QSettings setting("/sdcard/cfg/CFG_APPSet.ini");  // MQTT 配置文件
testIp = setting.value("General/testIp").toString(); // 读取 Broker 地址

// mqttsrv.cpp
test->tls_set(cafile, NULL, NULL, ...);     // 需要 TLS 证书
```

→ 需要创建 `/sdcard/cfg/CFG_APPSet.ini` 和 `/app/root.crt`。

### 4.2 配置字段为空导致静默跳过

```cpp
if (myHttpFileClient->g_connectName != "")
    test = new mqttsrv(g_connectName.toLocal8Bit());
// 如果 g_connectName 是空的，mqttsrv 根本不会创建
```

`g_connectName` 的来源是 `CFG_APPSet.ini` 里的 `connectName` 字段。配置文件里缺这个字段 → 整个 MQTT 静默不运行。**排查时不能只看有没有配置文件，还要看配置里的字段有没有值。**

### 4.3 .so 找不到

编译时链接成功（因为有 `-L` 指向 `3rdparty/mqtt/`），但运行时需要 `LD_LIBRARY_PATH` 告诉加载器去哪找。

```bash
ldd dist/zksort | grep "not found"
```

**MQTT 案例**：`libmosquittopp.so.1 => not found` → 需要在启动脚本加 `3rdparty/mqtt` 到 `LD_LIBRARY_PATH`。

### 4.4 TLS 证书

```bash
# 从 Broker 提取证书
openssl s_client -connect Broker地址:端口 2>/dev/null | \
    sed -n '/-----BEGIN CERTIFICATE-----/,/-----END CERTIFICATE-----/p' > /app/root.crt
```

**MQTT 案例**：`openssl s_client -connect cloud.chinaamd.cn:1884`

# 第二章：执行流程

## 2.1 诊断阶段（只查不改，全部找完再动手）

```
命令1 → 扫 #if 0:   grep -rn '^#if 0' zksort/ | grep -v build/
命令2 → 扫 #ifdef:  grep -rn '#ifdef Q_OS_UNIX' zksort/ --include='*.cpp' | grep -v build/
命令3 → 扫 //注释:  grep -rn '//.*new\|//.*start()' zksort/global/
命令4 → 扫 .pro:    grep -n '#.*-l\|#LIBS' zksort.pro
命令5 → 扫 库文件:  for d in 3rdparty/*/; do echo "$d: $(find $d -name '*.so' | wc -l)"; done
命令6 → 扫 运行时:  grep -rn '/sdcard/\|/app/' zksort/ --include='*.cpp' | grep -v build/
```

**每一步都记笔记，只记不改。画一张四层阻断图。**

## 2.2 执行阶段（严格按依赖顺序）

```
第1步 库文件就位    → 编译/下载匹配版本的 .so → 放到 3rdparty/对应目录
                      验证: file 3rdparty/xxx/*.so | grep x86-64
第2步 解封源码      → 删 #if 0 及配对 #endif → 删 //
                      验证: qmake && make（看编译是否通过）
第3步 恢复初始化    → 恢复 new/start() 调用
                      验证: nm dist/zksort | grep 目标类名（看符号是否在二进制里）
第4步 改 .pro       → 取消注释 LIBS 行
                      验证: ldd dist/zksort | grep 目标库（看运行时依赖是否齐全）
第5步 配运行时      → 创建配置文件 → 提取证书 → 更新 LD_LIBRARY_PATH
                      验证: timeout 5 ./dist/zksort（看是否崩溃）
```

## 2.3 为什么不能跳步

库没到位就解封源码 → 编译时 `fatal error: xxx.h not found`。初始化没恢复就配运行时 → 功能静默不生效，排查时找不到原因。

# 第三章：故障速查

| 症状 | 所在层 | 原因 |
|------|------|------|
| `fatal error: xxx.h: No such file` | 源码 | 头文件路径不在 INCLUDEPATH |
| `'XxxClass' was not declared` | 源码 | 类定义在 `#if 0` 里 |
| `#endif without #if` | 源码 | 删 `#if 0` 忘配对的 `#endif` |
| `undefined reference to 'xxx'` | 链接 | LIBS 被注释，或 .so 不存在 |
| `skipping incompatible` | 链接 | .so 架构不匹配（ARM vs x86_64） |
| `cannot find -lxxx` | 链接 | `-L` 目录下没有对应的 .so |
| 编译运行正常但功能不生效 | 初始化/运行时 | `// new` 被注释，或配置文件字段为空 |
| `error while loading shared libs` | 运行时 | LD_LIBRARY_PATH 缺目录 |
| `No such file or directory` | 运行时 | /sdcard 下缺配置文件 |
| 功能连不上 Broker | 运行时 | TLS 证书缺失，或 connectName 为空 |

# 第四章：工程管理

## 4.1 代码修改注释

```cpp
// [zcy 2026.06.15] mqtt config
// 为了适配 MQTT 的第三方支持和各项文件依赖所进行的改动
```

每个改动的文件顶部加此标记。改动集中提交：
```bash
git commit -m "mqtt config: 解封 MQTT 类定义和线程创建"
```

## 4.2 .gitignore

构建产物不入库：
```
zksort/build/
zksort/dist/zksort
3rdparty/*/build/
3rdparty/*/Makefile*
Makefile*
*.o
```

## 4.3 配置文档

每一次完整的功能配置，记录以下内容到 md 文档：
- 四层诊断结果（哪些控制点被禁用，如何发现的）
- 每步修改的具体内容和文件位置
- 排查过程中遇到的坑和解决方法
- 验证结果
