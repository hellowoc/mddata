## 一、交叉编译（§1.1-§1.2）

### Host vs Target

| 术语 | 含义 | 本工程中 |
|------|------|---------|
| Host（主机） | 运行编译器的那台机器 | x86_64 Ubuntu 虚拟机 |
| Target（目标） | 运行编译产物的那台机器 | ARM Linux 嵌入式板卡 |

当 Host ≠ Target 时，就需要交叉编译。

### 工具链（Toolchain）

一组协同工作的编译工具集合：

| 工具 | 功能 | 输入 → 输出 |
|------|------|-----------|
| 编译器 (gcc/g++) | 源码 → 目标文件 | .c/.cpp → .o |
| 汇编器 (as) | 汇编代码 → 目标文件 | .s → .o |
| 链接器 (ld) | 多个目标文件 + 库 → 可执行文件 | .o + .so → ELF 可执行文件 |
| 归档器 (ar) | 多个目标文件 → 静态库 | .o → .a |

### ARM 架构命名解析

`arm-linux-gnueabihf-gcc` 拆解：

```
arm       → CPU 架构（ARM）
linux     → 目标操作系统（Linux）
gnueabi   → GNU EABI（Embedded Application Binary Interface，嵌入式应用二进制接口）
hf        → Hard Float，ARM 芯片有硬件浮点单元（FPU）
```

没有 hf 后缀的 `arm-linux-gnueabi-gcc` 是软浮点版本——浮点运算用软件模拟。硬件浮点速度快 10 倍以上，本工程的图像处理涉及大量浮点运算，必须用 hf 版本。

### sysroot

交叉编译器需要 ARM 版的头文件和库文件。这些文件放在 sysroot 目录下，路径通常为 `/usr/arm-linux-gnueabihf/`。编译器编译时读这个目录下的头文件（如 `<sys/socket.h>`），链接时链接这个目录下的库文件（如 `libc.so`，不是 x86_64 的 libc）。

## 二、ELF 文件格式（§1.3-§1.5）

### ELF 是什么

Executable and Linkable Format。Linux 下可执行文件、目标文件(.o)、共享库(.so) 的统一格式。

### ELF 的内部结构

```
┌─────────────────────────────┐
│ ELF Header                  │  ← 魔数 0x7F 'E' 'L' 'F' / 架构(ARM/x86) / 入口地址
├─────────────────────────────┤
│ Program Header Table        │  ← 告诉内核：哪些段要加载到内存、加载到哪个地址
├─────────────────────────────┤
│ .text 段（代码段）            │  ← 只读，存放编译后的机器码
│ .rodata 段（只读数据）        │  ← 常量字符串、虚函数表等
├─────────────────────────────┤
│ .data 段（已初始化数据）       │  ← 全局变量初始值
│ .bss 段（未初始化数据）        │  ← 全局变量（初始值为0），ELF 中不占空间
├─────────────────────────────┤
│ .dynamic 段                  │  ← 动态链接信息（依赖哪些 .so）
│ .dynsym 段（动态符号表）       │  ← 导出/导入的函数和变量名
├─────────────────────────────┤
│ Section Header Table         │  ← 给链接器和调试器用的段描述，运行时不需要
└─────────────────────────────┘
```

### Name Mangling（名字修饰）

C++ 允许函数重载——同一个函数名可以有多个版本（参数不同）。为了在符号表中区分，编译器把函数名编码成唯一的符号：

```cpp
// 源码中
int mqttMsgParaseThread::onCmdFeed(string cmd_value)

// 编译后的符号名（GCC 的 mangling 规则）：
_ZN21mqttMsgParaseThread9onCmdFeedESs
// 拆解：
// _Z = C++ 符号前缀
// N = 嵌套名称开始
// 21mqttMsgParaseThread = 类名长度(21) + 类名
// 9onCmdFeed = 函数名长度(9) + 函数名
// E = 嵌套名称结束
// Ss = std::string 的 mangled 名
```

这就是为什么 nm 命令看 .o 文件会看到一堆奇怪的符号名。`extern "C"` 可以关闭 name mangling（C 语言没有重载），mosquitto 的 C API 就是加了这个的。

## 三、共享库加载（§2.2-§2.3）

### soname

Shared Object Name。每个 .so 文件在 ELF 头的 .dynamic 段中有一个 `SONAME` 字段。编译时链接器把这个 soname 写入可执行文件。运行时加载器按 soname 去找对应的 .so。

```bash
# 查看 .so 的 soname
readelf -d libmosquitto.so | grep SONAME
# 0x0000000e (SONAME)  Library soname: [libmosquitto.so.1]
```

可执行文件 zksort 的 .dynamic 段中：
```
NEEDED  libmosquitto.so.1    ← 需要这个 soname 的库
NEEDED  libmosquittopp.so.1
NEEDED  libQtCore.so.4
NEEDED  libssl.so.1.0.0
NEEDED  libcrypto.so.1.0.0
```

### ldconfig

```bash
ldconfig
# 扫描 /lib 和 /usr/lib（以及 /etc/ld.so.conf 中列出的目录）
# 为每个 .so 创建 soname → 文件路径 的映射
# 写入 /etc/ld.so.cache（二进制缓存文件）

ldconfig -p | grep mosquitto
# libmosquitto.so.1 (libc6, hard-float) => /lib/libmosquitto.so.1
# libmosquittopp.so.1 (libc6, hard-float) => /lib/libmosquittopp.so.1
```

### RPATH

编译时可以在可执行文件中写入 RPATH——运行时加载器优先搜索这个路径。这样 .so 可以放在 /app/lib 而不需要修改系统 /lib 目录。

```bash
# 编译时指定 RPATH
gcc ... -Wl,-rpath,/app/lib -o zksort

# 查看 RPATH
readelf -d zksort | grep RPATH
# 0x0000000f (RPATH)  Library rpath: [/app/lib]
```

### LD_LIBRARY_PATH

环境变量，运行时临时修改 .so 搜索路径。优先级最高。用于调试——测试新版本 .so 时不需要覆盖系统的库。

```bash
LD_LIBRARY_PATH=/tmp/test_libs ./zksort
# zksort 先从 /tmp/test_libs 找 .so，找不到再走 ld.so.cache
```

## 四、DNS 详解（§3.2）

### 域名层级

```
cloud.chinaamd.cn.
        │         │   │
        │         │   └── 顶级域名 (Top-Level Domain, TLD)
        │         └────── 二级域名
        └──────────────── 三级域名（子域名）

最后的 "." 是根域名，通常省略。"cn" 是中国国家顶级域名（ccTLD）
```

### DNS 解析的递归过程

```
1. 客户端 → 本地 DNS 服务器 (114.114.114.114):
   "cloud.chinaamd.cn 的 IP 是什么？"

2. 114.114.114.114 → 根 DNS 服务器 (.):
   "cn 的权威 DNS 服务器在哪？"
   根回答: "ns.cernet.net (202.112.0.44) 负责 .cn"

3. 114.114.114.114 → cn 权威服务器:
   "chinaamd.cn 的权威 DNS 服务器在哪？"
   cn 回答: "dns1.hichina.com (106.11.xx.xx) 负责 chinaamd.cn"

4. 114.114.114.114 → dns1.hichina.com:
   "cloud.chinaamd.cn 的 A 记录是什么？"
   dns1 回答: "120.26.xxx.xxx"

5. 114.114.114.114 → 客户端:
   "cloud.chinaamd.cn = 120.26.xxx.xxx, TTL=600 秒"

TTL = Time To Live：这条记录缓存 600 秒。600 秒内再查同一个域名，DNS 服务器直接返回缓存结果，不递归。
```

### DNS 记录类型

| 类型 | 全称 | 查询返回 |
|------|------|---------|
| A | Address | IPv4 地址 |
| AAAA | IPv6 Address | IPv6 地址 |
| CNAME | Canonical Name | 别名（一个域名指向另一个域名） |
| MX | Mail Exchange | 邮件服务器地址 |
| NS | Name Server | 该域名的权威 DNS 服务器 |

本工程只需要 A 记录——通过域名获取 IPv4 地址。

### /etc/resolv.conf

```bash
# 工业设备的常见配置
nameserver 114.114.114.114    # 国内通用 DNS
nameserver 8.8.8.8            # Google DNS（备用）
nameserver 192.168.1.1        # 工厂内网 DNS 服务器（如果工厂有）
```

如果 `resolv.conf` 为空或不存在，`getaddrinfo()` 只查 `/etc/hosts`。`connect()` 填的是域名而不是 IP 时，需要 DNS 或 hosts 提供 IP。

## 五、iptables 防火墙（§3.3）

### 三表五链

```
表（Table） — 按功能分类：

filter 表（默认表）：
  负责过滤数据包——允许还是拒绝
  INPUT 链：目标是本机的包（别人发给我的）
  OUTPUT 链：从本机发出的包（我发给别人的）
  FORWARD 链：经过本机转发的包（本机做路由器时）

nat 表：
  负责地址转换
  PREROUTING 链：包刚到达时修改目标地址（端口映射）
  POSTROUTING 链：包要离开时修改源地址（SNAT/Masquerade，共享上网）

mangle 表：
  负责修改包头（改 TTL、TOS 等），一般不用
```

### 常用命令

```bash
# 查看所有规则（带行号）
iptables -L -n -v --line-numbers

# 查看 nat 表
iptables -t nat -L -n

# 删除 filter 表 OUTPUT 链第 3 条规则
iptables -D OUTPUT 3

# 清空所有规则（危险，不要在生产环境随便用）
iptables -F

# 保存规则到文件（重启后丢失）
iptables-save > /etc/iptables.rules

# 恢复规则
iptables-restore < /etc/iptables.rules
```

### 对 MQTT 设备的影响

大部分嵌入式 Linux 固件默认 `policy ACCEPT`：所有出站连接允许。如果工厂网络的安全策略做了出站白名单，需要设备固件在内核配置中默认打开发出端口，或者在运维层面在路由器/防火墙上开端口。

## 六、NTP 时间同步（§3.4）

### 为什么 TLS 需要正确时间

```
证书有效期：
  Not Before: 2024-01-01 00:00:00 UTC
  Not After:  2034-01-01 00:00:00 UTC

如果系统时间是 1970-01-01 → 当前时间 < Not Before → 证书"还未生效"
如果系统时间是 2040-01-01 → 当前时间 > Not After → 证书"已过期"
两种情况 TLS 握手都会失败，返回证书验证错误
```

### ntpdate vs ntpd

```
ntpdate（本工程使用）：
  一次性同步。执行一次 → 获取当前时间 → 设置系统时间 → 退出
  缺点：如果网络波动导致获取失败，没有重试机制
  优点：不需要常驻后台进程，节省内存

ntpd（守护进程版本）：
  持续运行 → 定期同步 → 逐步调整系统时钟（避免时间跳变）
  缺点：占用内存（约 2-4 MB），增加系统负担
  优点：时间精确、平滑调整
```

### 硬件时钟和系统时钟

```
系统时钟（System Clock）：Linux 内核维护的时间，保存在内存中，重启丢失
硬件时钟（RTC, Real-Time Clock）：主板上的独立芯片，有电池，永久运行

系统启动时：内核从 RTC 读取初始时间
hwclock -w：把系统时钟写入 RTC（用于下次启动时读取） ← 本工程在 ntpdate 后执行
hwclock -s：把 RTC 时间读入系统时钟（启动脚本中自动执行）
```

## 七、init.d 与 systemd（§2.4）

### SysV init 启动流程

```
1. 内核启动 → 启动 init 进程（PID=1）
2. init 读取 /etc/inittab → 确定默认运行级别（通常是 5）

运行级别：
  0 = 关机
  1 = 单用户模式（维护用）
  2 = 多用户模式（无网络）
  3 = 多用户模式（有网络，命令行）
  5 = 多用户模式（有网络 + GUI）
  6 = 重启

3. init 执行 /etc/rc5.d/ 下的脚本：
   S01xxx → S02xxx → ... → S99xxx（按 S 后面的数字从小到大依次执行）
   K01xxx → K02xxx → ...（K 开头的在切换运行级别时执行，K = Kill 停止服务）

S99zksort → 实际是一个符号链接 → /etc/init.d/zksort start
"99" 确保 zksort 最后启动（网络、文件系统等都已就绪）
```

### systemd 启动流程

```
1. 内核启动 → 启动 systemd（PID=1）
2. systemd 读取所有 unit 文件：
   /etc/systemd/system/     ← 管理员自定义的（优先级最高）
   /usr/lib/systemd/system/  ← 软件包自带的
3. 按依赖关系并行启动：

target 之间的依赖：
  network.target
    ↑ After=network.target
  zksort.service
    ↑ After=network.target

systemd 确保 network.target 达成后才启动 zksort.service
并行启动无依赖关系的服务，比 init.d 的串行快
```

## 八、Makefile / qmake 基础

### qmake 的生成过程

```
.pro 文件（手写的）
  ↓ 运行 qmake
Makefile 文件（自动生成的，可以手动调，但不推荐）
  ↓ 运行 make
可执行文件
```

qmake 读 .pro 文件，根据项目和平台设置，生成对应的 Makefile。然后 make 执行 Makefile 中的编译和链接命令。

### Makefile 中的关键变量

```makefile
# 编译器
CC = arm-linux-gnueabihf-gcc
CXX = arm-linux-gnueabihf-g++

# 编译选项
CFLAGS = -O2 -Wall -I/path/to/include
# -O2：优化等级 2（平衡编译速度和运行速度）
# -Wall：开启所有警告

# 链接选项
LDFLAGS = -L/path/to/libs -lmosquitto -lssl -lcrypto

# 规则：
target: dependencies
	command

# 示例：
mqttsrv.o: mqttsrv.cpp mosquitto.h
	$(CXX) -c mqttsrv.cpp -o mqttsrv.o $(CFLAGS)
# 含义：如果 mqttsrv.cpp 或 mosquitto.h 的时间戳比 mqttsrv.o 新
#       就执行下面的命令重新编译。否则跳过。
```

## 九、RPATH/RUNPATH

### 两者的区别

```bash
# RPATH（rpath）：较老的标准
# - 优先级高于 LD_LIBRARY_PATH（正常情况下）
# - 影响所有间接依赖的搜索

# RUNPATH：较新的标准
# - 优先级低于 LD_LIBRARY_PATH
# - 只影响直接依赖的搜索，间接依赖用默认路径

# 查看：
readelf -d zksort | grep -E "RPATH|RUNPATH"
```

### 在 qmake 中设置

```qmake
# 方式 1：RPATH
QMAKE_LFLAGS += -Wl,-rpath,/app/lib

# 方式 2：RUNPATH
QMAKE_LFLAGS += -Wl,-rpath,/app/lib,--enable-new-dtags
# --enable-new-dtags 将 RPATH 替换为 RUNPATH
```

本工程没有设置 RPATH——依赖的 .so 放在系统默认路径 /lib 下，不需要自定义搜索路径。
