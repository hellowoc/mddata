## 第一章：编译与构建 — 从源码到可执行文件

### 1.1 问题的起点：开发机和目标机是两台不同的机器

你的开发机是 x86_64 的 Ubuntu 虚拟机，目标机是 ARM 架构的嵌入式 Linux 板卡。x86_64 的 CPU 只能执行 x86_64 指令集的机器码。ARM CPU 只能执行 ARM 指令集的机器码。如果在 Ubuntu 上用默认 GCC 编译，产生的是 x86_64 机器码——放到 ARM 板卡上运行，CPU 读到不认识的第一条指令，直接报 Illegal Instruction 崩溃。

交叉编译就是解决这个问题的：在一台机器上生成另一台机器的可执行文件。

这需要一整套交叉编译工具链。先看这个工具链里有什么，再回过头看怎么配置 qmake。

### 1.2 交叉编译工具链的内部结构

一个完整的 ARM Linux 交叉编译工具链包含：

```
arm-linux-gnueabihf-gcc          ← C 编译器（生成 ARM 机器码）
arm-linux-gnueabihf-g++          ← C++ 编译器
arm-linux-gnueabihf-ld           ← 链接器（把 .o 拼成可执行文件）
arm-linux-gnueabihf-as           ← 汇编器（把 .s 汇编文件变成 .o）
arm-linux-gnueabihf-objcopy      ← 格式转换工具
arm-linux-gnueabihf-strip        ← 符号表裁剪

以及一组 ARM 版的头文件和库文件（sysroot）：
  /usr/arm-linux-gnueabihf/include/  ← ARM 架构的 C 标准库头文件
  /usr/arm-linux-gnueabihf/lib/      ← ARM 架构的 .so 和 .a 文件
```

命名规则：`arm` = ARM 架构，`linux` = 目标操作系统，`gnueabihf` = GNU EABI 硬件浮点（hard float）。`hf` 表示 ARM 芯片有硬件 FPU，浮点运算用硬件指令而非软件模拟。

### 1.3 编译的四步过程（每条语句都在做什么）

以一行 `gcc -c mqttsrv.cpp -o mqttsrv.o` 为例，GCC 内部做了四件事：

```
============= 第一步：预处理（preprocessor）=============
输入: mqttsrv.cpp
处理:
  - #include "mosquitto.h" → 把 mosquitto.h 的全部内容（1500+ 行）原样插入
  - #ifdef Q_OS_UNIX → 条件编译，Linux 保留，Windows 删除
  - 宏展开 → MQTT_TEST → 1
输出: mqttsrv.i（预处理后的纯 C++ 源码，.cpp 文件会膨胀到数万行）

============= 第二步：编译（compile）=============
输入: mqttsrv.i
处理:
  - 词法分析：把源码拆成 token（int / mqttsrv / :: / run / ( / ) / { / ...）
  - 语法分析：把 token 序列构建成抽象语法树（AST）
    while_stmt
      ├── condition: (1)
      └── body: compound_stmt { ... }
  - 语义分析：检查类型匹配、变量作用域
  - 中间代码生成：把 AST 转成 GIMPLE（GCC 内部的中间表示，平台无关）
输出: GIMPLE IR（中间表示）

============= 第三步：汇编（assemble）=============
输入: GIMPLE IR
处理:
  - GIMPLE → RTL（Register Transfer Language，开始和具体 CPU 架构相关）
  - RTL → 汇编代码：
    对于 ARM：
      mov r0, #5        ← 把 5 装入寄存器 r0
      ldr r1, [sp, #8]  ← 从栈偏移 8 处加载到 r1
      bl  _ZN6mqttsrvC1Ev  ← 调用 mqttsrv 构造函数（name mangling 后的符号名）
    对于 x86_64：
      mov eax, 5
      mov rdi, [rsp+8]
      call mqttsrv::mqttsrv()
    同样的 C++ 代码，不同架构产生完全不同的汇编指令
输出: mqttsrv.s（ARM 汇编文件）→ mqttsrv.o（ARM 机器码的目标文件）

============= 第四步：链接（link）=============
输入: mqttsrv.o + globalflow.o + main.o + ... + libmosquitto.so + libQtCore.so + ...
处理:
  - 符号解析：mqttsrv.o 中调用了 mosquitto_connect() → 去 libmosquitto.so 里找
  - 重定位：每个 .o 文件内部地址都是"从 0 开始"，链接器把它们拼起来后重新计算实际地址
  - 把 ELF 头、代码段(.text)、数据段(.data)、符号表、动态链接信息写入最终文件
输出: zksort（ARM 可执行文件，ELF 格式）
```

### 1.4 qmake 工程中的 MQTT 配置

```qmake
# zksort.pro 中需要添加的配置：

# 1. 指定交叉编译器
QMAKE_CC  = arm-linux-gnueabihf-gcc
QMAKE_CXX = arm-linux-gnueabihf-g++
QMAKE_LINK = arm-linux-gnueabihf-g++

# 2. 添加头文件搜索路径（编译阶段用）
INCLUDEPATH += $$PWD/../3rdparty/mqtt/include
# 实际的头文件是 bus/mosquitto.h 和 bus/mosquittopp.h

# 3. 添加库搜索路径（链接阶段用）
LIBS += -L$$PWD/../3rdparty/mqtt
# -L 参数：告诉链接器"去这个目录找 .so 文件"

# 4. 添加要链接的库
LIBS += -lmosquitto -lmosquittopp
# -lmosquitto → 链接器去找 libmosquitto.so
# -lmosquittopp → 链接器去找 libmosquittopp.so

# 5. OpenSSL 依赖
LIBS += -lssl -lcrypto
# libmosquitto 依赖 OpenSSL 做 TLS
# 链接时如果不加这两行，链接器报错：undefined reference to SSL_CTX_new

# 6. 条件编译（只在 Linux 下链接 MQTT 相关库）
unix {
    LIBS += -lmosquitto -lmosquittopp -lssl -lcrypto
}
win32 {
    # Windows 下不链接 mosquitto，代码被 #ifdef Q_OS_UNIX 跳过
}
```

**为什么 `-L` 和 `-l` 是分开的**：`-L` 告诉链接器去哪里找库文件，`-l` 指定要链接哪些库。`-L` 可以有多个目录，链接器按顺序搜索。`-lmosquitto` 实际搜索 `libmosquitto.so`（动态链接）或 `libmosquitto.a`（静态链接，优先级高于 .so）。

### 1.5 动态链接 vs 静态链接

```
动态链接（本工程方式）：
  zksort 可执行文件
    依赖 → libmosquitto.so.1（运行时才加载到内存）
            libmosquittopp.so.1
            libssl.so
            libcrypto.so
  优点：可执行文件小、库可以独立更新、多个程序共享同一份 .so 节省内存
  缺点：部署时要保证目标机上有这些 .so 且版本匹配

静态链接：
  zksort 可执行文件（包含了 libmosquitto 的全部代码，文件体积大一倍以上）
  优点：一个文件，不用管目标机上有什么库
  缺点：库有安全漏洞时需要重新编译整个 zksort，不能只换 .so
```

本工程用动态链接——`3rdparty/mqtt/` 下放的是预编译的 `.so` 文件，zksort 编译时链接它们，部署时把 `.so` 一起复制到目标板。

### 1.6 预编译 .so 的来源

```bash
# 在 ARM 交叉编译环境下编译 mosquitto 1.4.14：

# 1. 下载 mosquitto 源码
wget https://mosquitto.org/files/source/mosquitto-1.4.14.tar.gz
tar xzf mosquitto-1.4.14.tar.gz
cd mosquitto-1.4.14

# 2. 配置交叉编译
CC=arm-linux-gnueabihf-gcc \
CXX=arm-linux-gnueabihf-g++ \
make WITH_TLS=yes WITH_TLS_PSK=no

# 3. 编译产物：
#    lib/libmosquitto.so.1       ← C 库
#    lib/cpp/libmosquittopp.so.1  ← C++ wrapper

# 4. 复制到工程
cp lib/libmosquitto.so.1 ~/CameraTest(aiuse)/3rdparty/mqtt/
cp lib/cpp/libmosquittopp.so.1 ~/CameraTest(aiuse)/3rdparty/mqtt/
```

## 第二章：文件系统部署

### 2.1 目标板卡的文件清单

```
设备出厂时需要预置以下文件：

/app/
  └── zksort                    ← 主可执行文件（编译产物）

/app/root.crt                   ← CA 根证书（PEM 格式，TLS 验证用）
                                 生成方式：云端运维提供或 openssl 命令生成

/sdcard/cfg/
  └── CFG_APPSet.ini           ← 配置文件，必须包含：
      [General]
      connectName=ZKGDDEV47     ← 设备唯一标识，MQTT Client ID
      testIp=cloud.chinaamd.cn  ← Broker 地址

/lib/ 或 /usr/lib/
  ├── libmosquitto.so.1         ← mosquitto C 库
  ├── libmosquittopp.so.1       ← mosquitto C++ wrapper
  ├── libssl.so.1.0.0           ← OpenSSL TLS 库
  ├── libcrypto.so.1.0.0        ← OpenSSL 加密库
  └── libmosquittopp.so         ← 符号链接 → libmosquittopp.so.1
      libmosquitto.so            ← 符号链接 → libmosquitto.so.1
```

### 2.2 .so 加载机制：系统怎么找到库文件

当 zksort 启动时，动态链接器（`/lib/ld-linux-armhf.so.3`）负责加载所有依赖的 .so。它按以下顺序搜索：

```
优先级从高到低：
1. LD_LIBRARY_PATH 环境变量指定的目录
2. /etc/ld.so.cache 缓存中记录的目录（由 ldconfig 生成）
3. /lib/ → /usr/lib/ → /usr/local/lib/（硬编码的默认路径）
4. ELF 文件中 RPATH/RUNPATH 指定的路径（编译时可以写死到可执行文件里）

# 本工程最常用的方式是方案 2：
ldconfig /lib /usr/lib    # 更新 .so 缓存
# 这个命令扫描 /lib 和 /usr/lib 下的 .so 文件，把它们的路径和 soname 写入 /etc/ld.so.cache
```

### 2.3 符号链接的必要性

```bash
# 实际文件
libmosquitto.so.1      ← 真实的库文件，soname 写在 ELF 头里

# 符号链接
libmosquitto.so        ← 链接器在编译 -lmosquitto 时查找这个文件名
libmosquitto.so → libmosquitto.so.1     # 链接器用的链接
libmosquitto.so.1 → libmosquitto.so.1.4.14  # 如果存在小版本

# 运行时加载器找的是 libmosquitto.so.1（soname），不是 libmosquitto.so
```

### 2.4 开机自启动

**方式一：init.d 脚本（本工程可能的方案，SysV init 系统）**

```bash
#!/bin/sh
# /etc/init.d/zksort

case "$1" in
  start)
    echo "Starting zksort..."
    cd /app
    ./zksort -qws &
    ;;
  stop)
    killall zksort
    ;;
  restart)
    $0 stop
    sleep 1
    $0 start
    ;;
esac

chmod +x /etc/init.d/zksort
update-rc.d zksort defaults  # 创建启动链接
# 这个命令在 /etc/rc5.d/ 下创建 S99zksort → ../init.d/zksort 的符号链接
# S = Start, 99 = 启动顺序（数字越大越晚启动）
# rc5.d 对应运行级别 5（多用户图形模式）
```

**方式二：systemd service（新版 Linux）**

```ini
# /etc/systemd/system/zksort.service
[Unit]
Description=ZKSort Color Sorter Application
After=network.target              # 等网络就绪后再启动
Wants=network.target

[Service]
Type=simple
ExecStart=/app/zksort -qws
WorkingDirectory=/app
Restart=always                    # 进程崩溃后自动重启
RestartSec=10                     # 等 10 秒再重启

[Install]
WantedBy=multi-user.target

# 启用：
systemctl enable zksort.service   # 开机自启
systemctl start zksort.service    # 立即启动
systemctl status zksort.service   # 查看状态
```

**`After=network.target` 的作用**：zksort 启动后 mqttThread 立即尝试连接 cloud.chinaamd.cn。如果网络还没就绪（eth0 没拿到 IP、路由表没配好），mqttThread 会在 ping 阶段失败退出。`After=network.target` 保证至少系统网络服务初始化完成后才启动 zksort。

## 第三章：网络要求

### 3.1 需求清单

| 需求                                | 用途                   | 不满足的后果                     |
| --------------------------------- | -------------------- | -------------------------- |
| DNS 解析 `cloud.chinaamd.cn`        | MQTT connect 需要目标 IP | `test->connect()` 失败       |
| 出站 TCP 到 `cloud.chinaamd.cn:1884` | MQTT TLS 连接          | 连不上 Broker                 |
| 出站 TCP 到 `cloud.chinaamd.cn:8900` | HTTP API（Token、上传）   | Token 获取失败，数据无法上传          |
| NTP 时间同步                          | TLS 证书有效期校验          | 系统时间可能被重置到 1970 年 → 证书验证失败 |
| 默认路由                              | 外网数据包知道从 eth0 出去     | ping 不通外网                  |

### 3.2 DNS 解析

```
当 test->connect("cloud.chinaamd.cn", 1884, 50) 被调用时，
内部发生了什么：

1. 应用程序：connect("cloud.chinaamd.cn", ...)
2. glibc：调用 getaddrinfo("cloud.chinaamd.cn")
3. /etc/nsswitch.conf：配置名称解析顺序
   hosts: files dns
   → 先查 /etc/hosts（files），再查 DNS
4. /etc/hosts 中没有 cloud.chinaamd.cn → 走 DNS
5. DNS 查询：读取 /etc/resolv.conf 中配置的 DNS 服务器地址
   nameserver 114.114.114.114
   nameserver 8.8.8.8
6. 向 114.114.114.114:53 发 UDP 查询包：
   "cloud.chinaamd.cn 的 A 记录是什么？"
7. DNS 服务器递归查询 → 返回 IP 地址，如 120.26.xxx.xxx
8. getaddrinfo() 返回 IP → connect() 拿到 IP → 发起 TCP 连接
```

**工业设备的两种 DNS 策略**：

```
策略 A：依赖外部 DNS 服务器（本工程默认）
  /etc/resolv.conf 配置 114.114.114.114 或公司内网 DNS
  优点：Broker 换 IP 不需要改设备
  缺点：DNS 服务器故障时 MQTT 连不上

策略 B：/etc/hosts 写死 IP
  echo "120.26.xxx.xxx cloud.chinaamd.cn" >> /etc/hosts
  优点：不依赖 DNS 服务器
  缺点：Broker 换 IP 需要 OTA 更新 hosts 文件
```

### 3.3 iptables 防火墙

Linux 内核的 netfilter 框架提供包过滤。iptables 是用户态管理工具。设备出厂前需要确认：

```bash
# 检查当前防火墙规则
iptables -L -n -v

# 允许出站 1884 端口（MQTT）
iptables -A OUTPUT -p tcp --dport 1884 -j ACCEPT

# 允许出站 8900 端口（HTTP）
iptables -A OUTPUT -p tcp --dport 8900 -j ACCEPT

# 允许出站 53 端口（DNS）
iptables -A OUTPUT -p udp --dport 53 -j ACCEPT

# 允许出站 123 端口（NTP）
iptables -A OUTPUT -p udp --dport 123 -j ACCEPT

# 查看默认策略
iptables -L OUTPUT
# Chain OUTPUT (policy ACCEPT) → 默认允许所有出站，通常不需要配 iptables
```

三表五链：
- filter 表（默认）：负责允许/拒绝（INPUT/OUTPUT/FORWARD 链）
- nat 表：负责地址转换
- mangle 表：负责修改包头

对 MQTT 设备来说，默认 OUTPUT 通常是 ACCEPT，不需要额外配置。但如果做了白名单加固，必须放开 1884、8900、53、123 四个端口。

### 3.4 NTP 时间同步

TLS 证书有有效期（Not Before ~ Not After）。如果设备系统时间被重置到 1970 年 1 月 1 日（RTC 电池没电、系统异常重启），当前时间远早于证书的 Not Before → TLS 握手拒绝证书 → MQTT 永久连不上。

本工程处理：

```cpp
// mqttsrv.cpp:151-152
system("ntpdate 0.pool.ntp.org");  // 网络时间同步
system("hwclock -w");               // 把系统时间写入硬件 RTC 芯片
```

这两个命令在 mqttThread 连接 MQTT 成功后执行的。如果 NTP 同步失败（DNS 没通、NTP 服务器不可达），设备继续用旧的系统时间——可能 TLS 验证失败，也可能正常（如果时间偏差不大）。

**生产环境改进建议**：在 NTP 同步成功后记录最后同步时间到 /sdcard/cfg/last_ntp_ok；程序启动时如果系统时间 < 2020 年 1 月 1 日且 last_ntp_ok 为有效时间戳，用该时间戳临时修正系统时间。

## 第四章：安全部署

### 4.1 root.crt 的管理

```
当前方案：root.crt 随固件预置在 /app/root.crt
问题：如果 CA 证书到期（通常 10-20 年），设备无法连接 Broker
     如果要切换 CA，所有设备需要 OTA 更新 root.crt

改进方案：
  1. root.crt 存储在 /sdcard/cfg/root.crt（可 OTA 更新）
  2. 固件中保留一个"工厂 root.crt"作为备份
  3. 代码中先读 /sdcard/cfg/root.crt，找不到再读 /app/root.crt
```

### 4.2 client_secret 存储

```cpp
// 当前在代码中硬编码：
// myhttpfileclient.cpp
"Basic YXBpOmFwaQ=="  → Base64 解码 → "api:api"

// 安全风险：
//   反编译 zksort 可执行文件 → 看到 "YXBpOmFwaQ==" → 被盗用

// 改进方案：
//   client_secret 存储到 CFG_APPSet.ini 的 [Security] 节
//   出厂时每台设备生成独立的 client_secret
```

### 4.3 设备标识的不可篡改

```
connectName = "ZKGDDEV47" 是设备在云端的身份
如果被篡改 → 可以冒充别的设备收走命令

当前存储：CFG_APPSet.ini（文本文件，可被修改）
改进方案：connectName 写入 /sys/class/efi/efivars 或加密存储
          出厂时一次性烧录，不允许修改
```

### 5.3 日志诊断

本工程 MQTT 相关日志输出点：

```cpp
// mqttsrv.cpp — 连接日志
qDebug()<<"iot addr : "<<testip;
qDebug()<<"connect ok";
qDebug()<<"MQTT连接成功";

logger()->info("reconnect");                     // 重连

// mqttsrv.cpp — 消息日志
qDebug()<<"来自<"<< topic <<">的消息："<< strRcv;

// mqttsrv.cpp — 发布日志
qDebug()<<"onCmdReturn errcode"<<errorcode;
```

部署后可以监控 `/sdcard/logs/` 或 `dmesg` / `journalctl`，确认：
1. "MQTT连接成功" — TLS 握手和 MQTT CONNECT 都成功
2. "订阅 mid = N" — SUBSCRIBE 成功
3. 如果频繁出现 "reconnect" — 网络不稳定或 Broker 端有问题

## 第六章：跨平台编译 — Windows Mock / Linux 真实连接

### 6.1 为什么需要跨平台

开发阶段不可能每次都交叉编译、烧录到 ARM 板卡测试。在 Windows 上编译调试能加快迭代速度。但 MQTT 依赖 Linux 原生库（mosquitto、OpenSSL、BSD Socket）。Windows 上没有这些。

### 6.2 条件编译策略

```cpp
// mqttsrv.h:77
#ifdef Q_OS_UNIX
class mqttsrv : public mosqpp::mosquittopp { ... };
class mqttThread : public QThread { ... };
class mqttMsgParaseThread : public QThread { ... };

extern mqttThread *myMqttThread;
extern mqttMsgParaseThread *myMqttMsgParaseThread;
#endif

// mqttsrv.cpp:5-7
#ifdef Q_OS_UNIX
mqttThread *myMqttThread;
mqttMsgParaseThread *myMqttMsgParaseThread;
#endif

// globalflow.cpp:10057-10066
#ifdef Q_OS_UNIX
    myHttpFileClient = new MyHttpFileClient;
    myMqttThread = new mqttThread;
    myMqttMsgParaseThread = new mqttMsgParaseThread;
    if (ipForInterface == "162.254.129.100") {
        myMqttThread->start();
        myMqttMsgParaseThread->start();
    }
#endif
```

Windows 编译时：`Q_OS_UNIX` 未定义 → 三件套的类定义、全局指针、创建和启动代码全部跳过。MQTT 相关模块在 Windows 上完全不存在，不会产生编译错误。

### 6.3 zksort.pro 中的条件配置

```qmake
# 只在 Linux 下链接 mosquitto
unix {
    LIBS += -L$$PWD/../3rdparty/mqtt -lmosquitto -lmosquittopp
    LIBS += -lssl -lcrypto
}

# Windows 下只链接 Qt 核心库，不碰 MQTT
win32 {
    # 什么都不加
}
```
