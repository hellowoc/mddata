# 签名

```
============= 签名（发件人） =============

1. 对要签名的消息算哈希值（摘要）
   消息: "CMD_Feed:50:20:30:80"
   SHA256 → 7465f2a9b3... (32字节)

2. 用自己的私钥对这个哈希值加密
   哈希值 + 私钥 → RSA 加密 → 签名值 S (256字节)

3. 把 (消息原文 + 签名值 S) 一起发出去


============= 验证（接收人） =============

4. 对收到的消息原文算哈希值
   消息: "CMD_Feed:50:20:30:80"
   SHA256 → 哈希值 H1 = 7465f2a9b3...

5. 用发件人的公钥解密签名值
   签名值 S + 公钥 → RSA 解密 → 哈希值 H2 = 7465f2a9b3...

6. 比较 H1 == H2 ?
   == → 两个事实同时成立：
       ① 消息没有被篡改过（哪怕改一个bit，SHA256输出就完全不同）
       ② 消息确实来自私钥持有者（别人没有私钥，签不出能被公钥解开的签名）

```

## 为什么要先哈希再签名，不直接对消息签名

消息可能很长（比如几 MB 的文件）

```
直接签名：对整个几 MB 跑 RSA 加密 → 极慢（RSA 比对称加密慢 100-1000 倍）

先哈希再签名：
  消息 → SHA256 → 32 字节的哈希值（固定长度，无论原消息多大）
  然后只对这个 32 字节跑 RSA → 快到几乎不耗时

```
## 数字签名在 TLS 证书验证中的作用

回到已看过的本工程：

```
cloud.chinaamd.cn 服务器发来的证书里有一个 Signature Value
这个值就是 CA 用自己的私钥对证书内容做的签名

你的工控机验证时：
  1. root.crt 里有 CA 的公钥
  2. 对证书的内容部分重新算哈希
  3. 用 CA 公钥解密 Signature Value，得到原始哈希
  4. 两个哈希对比 → 相等 → 证书确实由该 CA 签发，内容没被篡改

如果黑客伪造了一个证书：
  - 他改了域名（改成 cloud.chinaamd.cn）
  - 但他没有 CA 的私钥 → 签不出正确的 Signature Value
  - 你验证时 H1 != H2 → 拒绝连接
```

这就是整个 TLS 信任链的基础——不是信"这个域名是谁"，而是信"我这个 root.crt 对应的 CA，只给合法域名签发证书，CA 的私钥没泄露"。

## 整个过程

```
============= 签名 =============

原始数据: "CMD_Feed:50:20:30"
  ↓ SHA256（这是固定算法，不涉及任何密钥）
哈希值 H = 0x7a3f...（32 字节）

签名: 把 H 和私钥(d, n) 放进 RSA 算法做一次数学运算：
  S = H^d mod n
  S = 0xb2e4...（256 字节）


============= 传输 =============

发出: (原始数据 "CMD_Feed:50:20:30", 签名值 S)


============= 验证 =============

收到原始数据 "CMD_Feed:50:20:30"
  ↓ SHA256（同一个固定算法）
H1 = 0x7a3f...

收到的签名值 S → 用公钥(e, n) 做数学运算：
H2 = S^e mod n

代进去：H2 = (H^d)^e mod n = H^(d×e) mod n

因为 e 和 d 是数学上配对选出来的——满足 e×d ≡ 1 (mod φ(n))
所以 H^(d×e) mod n = H mod n = H

H2 = H = H1 = 0x7a3f...
```
## Q通过s验证证书是真的，然后根据证书内容协商出来加密方式，然后用这个加密方式来传输数据

1. **加密方式**（用哪个算法）——握手前几句就定好了，不依赖证书。客户端说"我支持 AES-256 / AES-128"，服务器选一个。
    
2. **会话密钥**（具体用哪把钥匙）——依赖证书中的服务器公钥。客户端生成 pre_master_secret，用服务器公钥加密传给服务器，双方各自算出相同的会话密钥。
    

证书的作用：验证服务器身份 → 确保你协商的对象是真的服务器 → 确保你加密传给它的 pre_master_secret 只有真正的服务器能解开。


## Q协商的过程中的双方通信也是通过非对称加密吗，此时还没有商量出来加密方式

```
1. ClientHello                             ← 明文
   "我支持 TLSv1、AES-256、RSA..."
   没有秘密，就是报菜名

2. ServerHello                             ← 明文
   "好，我们用 TLSv1 + AES-256 + RSA"
   加密方式在这一步就商量好了

3. Certificate                             ← 明文
   证书本身就是公开文件，不需要加密
   证书里的签名 S 才是真伪验证的关键

4. ServerHelloDone                         ← 明文
   "我说完了"

5. ClientKeyExchange                       ← ★ 唯一用非对称加密的
   pre_master_secret 用服务器公钥加密后发出
   只有服务器私钥能解开

6. ChangeCipherSpec                        ← 明文
   "从现在开始，下面的内容用会话密钥加密"

7. Finished（加密的）                       ← ★ 从这里开始对称加密
8. ChangeCipherSpec（加密的）
9. Finished（加密的）
```


加密方式（算法）**在第 2 步就确定了，不需要保密——AES-256 算法本身是公开的。

**会话密钥**在第 5 步用非对称加密安全传输，只有服务器能解开。

从第 7 步开始，所有后续消息都用这个会话密钥做对称加密。


TLS 只做了**单向认证**——客户端验证服务器。服务器没有验证客户端。因为本工程没有配置客户端证书（`tls_set` 的后三个参数全是 NULL）。

那服务器怎么知道这条 TLS 连接是设备 ZKGDDEV47 而不是某个冒充的？**OAuth2 Token 填补了这个缺口。**

```
设备启动 → 向云端申请 Token：
  POST /auth/mobile/token/device
  Authorization: Basic YXBpOmFwaQ==  (client_id:client_secret 的 Base64)
  Body: mobile=DEVICE@ZKGDDEV47

云端返回：
  { "access_token": "eyJhbGciOi...", "expires_in": 604800 }

之后所有 HTTP 请求都带上这个 token：
  POST /api/upload
  Authorization: Bearer eyJhbGciOi...
  
云端收到后验证 token → 确认是 ZKGDDEV47 → 允许操作
```

## MQTT + TLS + OAuth2 三者的分工

| 层      | 解决什么问题         | 本工程的实现                                       |
| ------ | -------------- | -------------------------------------------- |
| TLS    | 传输加密 + 验证服务器身份 | `tls_set("./root.crt")` — 确认对面是真的 Broker     |
| MQTT   | 消息的发布/订阅       | 手机发命令到 `/ZKGDDEV47`，设备回传到 `MqttServerClient` |
| OAuth2 | 验证客户端身份        | Token 证明"我是 ZKGDDEV47"，不是伪造设备                |

**一个没有 OAuth2 的后果**：攻击者在内网搭一个假设备，用同样的 connectName 连上 Broker，订阅 `/ZKGDDEV47` — 所有发给真设备的命令被假设备收走。OAuth2 的 Token 让云端能区分"谁来申请 Token 时提供了正确的 client_secret"。
