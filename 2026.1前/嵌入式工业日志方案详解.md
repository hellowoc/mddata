# 嵌入式工业日志方案详解

---

## 一、嵌入式日志的核心难题

| 难题 | 学术项目 | 工业设备 |
|------|---------|---------|
| 存储空间 | 几乎不限制 | SD 卡 4~16GB，要存数月 |
| 断电 | 不存在（随手关） | 随时可能被拉闸 |
| 写入寿命 | 不用考虑 | NAND Flash 有擦写次数上限 |
| 事后排查 | 复现 bug 即可 | 设备在千里之外的工厂，只能靠日志 |
| 性能影响 | 无所谓 | 1ms 的日志延迟可能影响分选精度 |
| 网络 | 通常有 | 很多设备完全离线 |

---

## 二、六种主流方案

### 方案一：syslog / rsyslog（Linux 标准守护进程）

```
应用程序 → syslog() 系统调用 → syslogd 守护进程 → /var/log/messages
```

**优点：**
- Linux 标配，零开发成本
- 支持远程转发（`@192.168.1.1:514`）
- 按 facility/priority 分类过滤

**缺点：**
- 对嵌入式而言太重（守护进程常驻内存）
- 日志格式固定，不易定制
- 默认路径 `/var/log` 在只读 rootfs 上写不进去
- 不支持按大小/时间自动切割（需要配 logrotate）

**适用场景：** 有完整 Linux 发行版的设备（如基于 Debian 的网关设备）

---

### 方案二：应用层自写文件 + 手动轮转（CameraTest 项目的方案）

```
应用程序 → 自己 open/write → /sdcard/logs/log.txt
         → 启动时检查 → 按规则重命名/删除
```

**优点：**
- 完全掌控，无外部依赖
- 实现简单，几百行代码
- 可以自定义日志格式、轮转策略

**缺点：**
- 每个项目都要重复造轮子
- 并发写需要自己做锁
- 没有现成的日志级别过滤、远程转发

**适用场景：** 资源受限的单片机/ARM 设备，CameraTest 就是这类

---

### 方案三：Log4j 风格框架（Log4Qt / Log4cxx / spdlog）

```
应用程序 → Logger::info() → Appender → 文件/控制台/网络
```

CameraTest 用了 Log4Qt（Java log4j 的 Qt 移植版）。

**优点：**
- 成熟的架构（Logger → Appender → Layout 三级分离）
- 支持运行时不重启调整日志级别
- 多种 Appender（文件、滚动文件、网络、数据库）
- 格式化灵活（PatternLayout）

**缺点：**
- 引入额外库依赖
- 框架本身有一定内存开销
- 嵌入式交叉编译时需要额外配置

**适用场景：** 中大型嵌入式 Qt/C++ 项目

---

### 方案四：内存环形缓冲区 + 定时刷盘

```
应用程序 → 写入内存 ring buffer（如 64KB 循环覆盖）
         → 定时器每 30 秒刷一次到文件
         → 崩溃时通过信号处理器紧急刷盘
```

**优点：**
- **写磁盘频率极低**，Flash 寿命大幅提升
- 性能极高（内存写 vs 磁盘写差距 1000 倍）
- 断电最多丢 30 秒日志（可接受）

**缺点：**
- 实现复杂（需要自己管理 ring buffer）
- 崩溃时可能丢失缓冲区中的日志
- 不适合需要实时查看日志的场景

**适用场景：** 高频日志场景（如每条物料 10 条日志 × 每秒 1000 条物料）

---

### 方案五：数据库存储（SQLite）

```
应用程序 → INSERT INTO log_table → SQLite → .db 文件
```

CameraTest 项目中部分统计功能用了 SQLite。

**优点：**
- 结构化查询（`SELECT * FROM log WHERE level='ERROR' AND time > '2026-05-01'`）
- 事务保证一致性（断电不丢数据）
- 支持索引（按时间、级别快速检索）

**缺点：**
- 写入性能远低于直接写文件
- `.db` 文件会不断增长，需要手动 VACUUM
- 数据库文件损坏后整个日志表丢失（不像文本文件部分可读）

**适用场景：** 需要结构化查询的统计日志（如产量统计），不适合高频实时日志

---

### 方案六：远程日志（MQTT / HTTP / gRPC）

```
应用程序 → MQTT publish → Broker → 云端存储
```

CameraTest 项目中的 MQTT 模块可以上报日志到远程。

**优点：**
- 本地不占空间
- 可以在线实时监控所有设备
- 日志不会因设备损坏而丢失

**缺点：**
- **依赖网络**（离线设备完全不可用）
- 网络延迟可能影响实时性
- 需要额外的云端基础设施

**适用场景：** 联网设备 + 远程运维平台

---

### 方案对比总表

| | syslog | 自写文件 | Log4Qt | 内存缓冲 | SQLite | 远程 |
|---|---|---|---|---|---|---|
| 开发成本 | 零 | 中 | 低 | 高 | 低 | 高 |
| 运行开销 | 中 | 低 | 低 | 极低 | 中 | 中 |
| Flash 寿命 | 差 | 差 | 差 | **优** | 中 | **优** |
| 断电安全 | 中 | 差 | 差 | **优** | **优** | **优** |
| 离线可用 | 是 | 是 | 是 | 是 | 是 | **否** |
| 查询便利 | 差 | 差 | 差 | 差 | **优** | 中 |
| 适合场景 | 通用 Linux | 简单设备 | Qt/C++ 项目 | 高频日志 | 统计查询 | 联网设备 |

## 三、CameraTest 项目的具体做法

这个项目是"**方案二 + 方案三**"的混合体：

### 启动阶段的纯手工管理（方案二）

```cpp
// main.cpp —— 不依赖框架，直接操作文件

// 1. 日志数量上限：最多 30 个文件
if (fileInfo.size() >= 30) {
    QFile::remove(fileInfo.at(0).absoluteFilePath());  // 删最老的
}

// 2. 跨天迁移：昨天的日志归档
if (sections.at(0) != dateString) {
    copyFileToPath(logFileInfo, logFileInfo + "." + sections.at(0));
    delFile(logFileInfo);                              // log.txt → log.txt.2026-05-19
}
```

### 运行阶段的 Log4Qt 框架（方案三）

```cpp
// main.cpp
Log4Qt::PropertyConfigurator::configure(qtlogProper);

// 业务代码中：
logger()->info("power on: %1 times", onoff_cnts);
logger()->warn("alarm turnoff feed");
logger()->error("save json end fail");
```

### 混合架构的实际效果

```
日志输出路径: /sdcard/logs/log.txt
日志框架:     Log4Qt (Logger → DailyRollingFileAppender → PatternLayout)
轮转策略:     手动（启动时检查跨天 + 数量上限）
格式:         "%d [%t] %p %c%x - %m%n"
              → "2026-05-21 14:30:00 [Mainthread] INFO ZKSort - power on: 1523 times"
级别:         DEBUG
容量控制:     最多 30 个归档文件
```

## 四、为什么工业设备要这样设计

### 问题：为什么不直接用 syslog

```
syslogd 守护进程 → 内存占用 ~500KB
在 256MB RAM 的设备上，500KB 不是小数
况且 syslog 默认写 /var/log，这个路径在只读的 squashfs 上
```

### 问题：为什么不在运行时轮转

```
运行时轮转 = 程序运行中检查日志大小 → 达到阈值就重命名
这需要另一个线程持续监控 → 多了线程切换开销
在 1ms 级实时控制系统中，能少一个线程是一个
启动时一次性检查是最安全的方式——只影响启动时间
```

### 问题：为什么是 30 个文件

```
假设每天一个日志文件，每个文件约 1MB：
30 个 = 30MB = 保留一个月的历史
SD 卡可用空间约 4-8GB，30MB 只占不到 1%
如果需要追溯超过一个月的 bug，通常说明有其他问题
```

### 问题：为什么不清零文件而是删除

```
文件系统删除 = 释放 inode → 空间立即可用
清零文件 = 先 open → write 0 → close → 可能留下空文件碎片
直接删除最干净，性能也最高（只改 inode 位图，不动数据块）
```

## 五、工业日志的黄金法则

### 1. 日志要能回答"设备当时在干什么"

```cpp
// 差的日志
logger()->info("value changed");

// 好的日志
logger()->info("fpga mode switched from %1 to %2, triggered by chute %3",
               oldMode, newMode, chuteId);
```

每行日志要让一个没见过这个设备的人，能看懂发生了什么。

### 2. 关键状态变更必须落日志

```
设备启动/关机
配置保存/加载
通信连接/断开
报警触发/消除
固件升级开始/结束
用户登录/登出（权限变更）
```

### 3. 日志级别要有纪律

```cpp
logger()->debug()  // 开发调试，量产版本关闭
logger()->info()   // 正常业务流程（开机、通信建立、配置保存成功）
logger()->warn()   // 异常但能自愈（串口超时重试成功、温度偏高但未超阈值）
logger()->error()  // 需要人工介入（FPGA 通信失败、配置保存失败）
logger()->fatal()  // 导致程序退出的错误（CameraTest 中没用这个级别）
```

### 4. 每行日志要有时间戳 + 线程标识

```
// CameraTest 的格式
2026-05-21 14:30:00.123 [Mainthread] INFO  ZKSort    - power on: 1523 times
                        ^^^^^^^^^^^^
                        没有这个，多线程 bug 完全无法排查
```

### 5. 断电安全策略

```cpp
// CameraTest 的做法 —— 每次关键操作后 sync
g_Runtime().mySystem("sync");   // 在 save()、cmd.sh 执行后

// Log4Qt 的配置
immediateFlush = true            // 每条日志立刻 flush 到磁盘
```

`immediateFlush = true` 对 Flash 寿命不友好，但在"断电丢日志 = 无法排查 bug = 客户退货"的权衡下，宁可牺牲 Flash 也要保证日志完整。

如果要兼顾：可以 `immediateFlush = false` + 每 5 秒定时 sync → 断电最多丢 5 秒日志。

### 6. Flash 磨损均衡

```
NAND Flash 擦写次数：通常 10 万次（MLC）/ 100 万次（SLC）

坏习惯：
  每条日志都 immediateFlush → 每秒 100 次写入 → 同一扇区快速磨损

好习惯：
  日志写到单独的分区（如 /sdcard/logs/）
  使用支持 wear-leveling 的文件系统（如 UBIFS、F2FS）
  日志文件达到一定大小后创建新文件（而非在同一个文件上覆盖）
```

CameraTest 使用按天滚动（DailyRollingFileAppender），每天写一个新文件，昨天的文件不再被修改，这天然避免了在同一扇区反复擦写。

### 7. 日志不要影响正常功能

```cpp
// 错误做法：在 1ms 定时器回调里写磁盘日志
void onTimerTick() {
    logger()->info("sensor value: %1", sensorValue);  // 磁盘 I/O 可能耗时 10ms+
    // 定时器已经丢了好几拍了
}

// 正确做法：只记到内存，定时刷盘
void onTimerTick() {
    ringBuffer.append("sensor value: xxx");   // 纯内存操作，< 1μs
}
```

## 六、总结：方案选择决策树

```
                     需要远程监控吗？
                    /               \
                  是                 否
                   ↓                  ↓
              MQTT/HTTP          Flash 磨损是问题吗？
              远程日志            /              \
                                是               否
                                 ↓                ↓
                          内存缓冲+定时刷盘     存储空间够吗？
                                                 /          \
                                               是           否
                                                ↓            ↓
                                         Log4Qt/自写     SQLite
                                         文件+轮转      (方便查)
```

CameraTest 走的是右下角路径：Log4Qt + 启动时手动轮转，适合离线/空间充足/对 Flash 寿命不极度敏感的场景。
