## 一、你的理解纠错与补充

你的描述基本正确，以下几点补充/修正：

| 你的描述 | 修正/补充 |
|---------|----------|
| "在sort类创建的时候就存在了一个定时器" | 定时器在 ZKSort 构造函数末尾创建，100ms 后触发，不是"创建时就存在"——是有一个 100ms 延迟 |
| "第一个处理的是加密器验证任务" | 正确。`mytimer → dccryptCheck()` 是 exec() 后第一个业务代码 |
| "信息的显示通过mylanauge直接获得不同语言的内容" | 是 `g_myLan()`（MyLanguage 单例），不是 mylanguage。从 .ini 文件读取翻译 |
| "整个工程的数据通过data在上位机适配和下位机适配的数据格式中完成转化" | data/ 层只做 C++ 结构体 ↔ JSON 文件转换，**不参与上下位机通信**。上下位机通信由 GlobalFlow + bus/ 完成 |
| "globalflow在后台将上位机数据下发" | GlobalFlow **不在后台**——它在主线程中被调用。后台的是 bus/ 层的各种通信线程 |
| "通过串口获得下位机的全局参数" | 正确，但不仅串口——还有 TCP（图像）、UDP（IPC 相机）、MQTT（云） |

## 三、以大米分选为例的完整业务流程

### 阶段 1：开机

```
设备上电
    │
    ▼
main() 启动
  ├── 创建崩溃转储目录（./crash/）
  ├── 初始化 Qt 应用（zkApplication，注册背光管理）
  ├── 设置 UTF-8 编码
  ├── 检查 U 盘触发文件（升级/清配置/执行脚本）
  ├── 显示 boot.bmp 启动画面
  ├── 日志管理（跨天迁移 + 数量控制，< 10ms）
  ├── 配置 Log4Qt（日志输出到 /sdcard/logs/log.txt）
  ├── 注册 resource.rcc 资源包
  ├── myFlow.initAll()
  │     ├── 读 JSON 配置 → 填充 struCnfg（机器型号：大米机、层数:2、通道:64...）
  │     ├── 读 JSON 方案 → 填充 struCnfp（当前方案：灰度灵敏度50、SVM 系数...）
  │     ├── 打开串口 1（连接 FPGA/相机板/接口板）和串口 2（连接控制板）
  │     ├── 创建并启动通信线程（2 发送 + 2 接收）
  │     └── 创建 updateStatusThread（轮询硬件状态）
  ├── 加载语言（上次用户选的，默认中文）
  ├── 创建 ZKSort 主窗口（全屏 1024×768）
  │     ├── 加载 QSS 皮肤
  │     ├── 交接 QStackedWidget 给 MiddleWidgetManager
  │     ├── 记录开机次数到 QSettings
  │     └── 启动 100ms 定时器
  └── a.exec() 进入事件循环
```

### 阶段 2：加密校验 + 硬件握手

```
100ms 定时器到期
    │
    ▼
dccryptCheck()
  ├── 检查加密授权（从 EEPROM 读取密码，用 time_encrypt 库解密）
  ├── 校验通过 → communication()
  │     ├── 显示"通信中，请稍候..."
  │     ├── myFlow.resetCommunication()
  │     │     ├── 遍历所有层/视/单元
  │     │     ├── 串口发查询指令（CMD_ALL_VERSION_REQUIRE）
  │     │     ├── 等待回包
  │     │     └── 更新 struGsh.struVer（各板卡版本和在线状态）
  │     ├── 通信成功 → startEnterMainPage()
  │     │     ├── myFlow.initSendAllParams()
  │     │     │     → 依次下发：机型配置 → 像素划分 → 模拟增益 → 偏置
  │     │     │     → 背景设置 → 算法参数 → 喷阀时间 → 光源亮度 → 供料量...
  │     │     │     （每个参数都是: 读结构体 → 组数据包 → 串口发送 → 等待确认）
  │     │     ├── 等 IPC 模型加载（最多 15 秒）
  │     │     ├── CreateWidgets() → 创建主页面
  │     │     └── ShowWidget(SM_MAIN_PAGE_WIDGET_NEW) → 显示主页
  │     └── 通信失败 → ShowWidget(SM_SELF_CHECK_WIDGET) → 自检页
  │
  └── dccryptCheck 末尾：
        connect(myUpdateStatusThread, SIGNAL(sAlarmTurnOnSig()), ...)
        connect(myUpdateStatusThread, SIGNAL(sWipeStartSig()), ...)
        ... (接好所有后台线程信号)
```

### 阶段 3：用户操作 — 开始分选大米

```
用户在主页面点 [运行]
    │
    ▼
NewMainPage::on_runBtn_clicked()
  ├── 检查 IPC 模型是否加载完（allIpcModelLoadOk == 2）
  ├── 检查报警状态（气压是否正常、料位是否正常...）
  ├── 弹出确认框"开始分选？"
  └── 用户确认 → myFlow.onOff()
        │
        ├── (1) 启动供料：myFlow.resetFeederRG()
        │       → 读 struCnfp.struGroupCtrl[...].nFeederValueMain[]（供料量）
        │       → 组包 CMD_CTRL_FEED_SET → 串口2 → 控制板 → 振动器启动
        │       大米从料斗落到振动槽，均匀排列进入分选通道
        │
        ├── (2) 设置光源：myFlow.resetLampRG()
        │       → 读 struCnfp.nLampValue[][]（灯亮度）
        │       → 组包 CMD_LAMP_BRIGHT → 串口 → 灯板 → 灯光亮起
        │       大米在光照下呈现不同颜色（好米透白、坏米暗黄/发黑）
        │
        ├── (3) 启动相机：cameraWaveCaliEnable()
        │       → 相机开始连续采集，每行 1024 像素
        │       → 通过 TCP 将图像数据发送到上位机
        │
        ├── (4) 启动 IPC AI 检测：myFlow.configIpcSortStart(1)
        │       → UDP 指令 → IPC 智能相机
        │       → IPC 加载 AI 模型（如 yolov5）
        │       → 开始推理：每粒大米 → AI 分类（好米/碎米/异色/霉变...）
        │
        ├── (5) 下发分选参数：myFlow.resetSortOnOff(1, 0)
        │       → FPGA 收到所有算法参数（灰度阈值、SVM 系数、HSV 范围...）
        │       → FPGA 开始实时处理每行 1024 像素
        │
        └── (6) 开启喷阀：myFlow.vavleOnOff()
                → 接口板准备工作，等待 FPGA 发出喷阀信号
    │
    ▼
分选运行中（持续循环）:
    │
    ├── 大米下落 → 相机线扫描 → 图像数据
    │       │
    │       ├── 传统 FPGA 方案：
    │       │     FPGA 接收原始像素 → 应用灰度算法（亮度阈值）
    │       │     → 应用 SVM 算法（多维特征分类）
    │       │     → 应用 HSV 算法（颜色空间判断）
    │       │     → 判断每粒大米：好 / 坏
    │       │     → 坏米到达喷阀位置 → 发喷气指令
    │       │     → 高压气喷出 → 坏米偏离轨道 → 落入坏料斗
    │       │     → 好米自由落体 → 落入好料斗
    │       │
    │       └── IPC AI 方案（可替代或辅助 FPGA）：
    │             IPC 相机内置 AI 推理 → 52 个分类 → 结果发回上位机
    │
    ├── 同时，updateStatusThread 在后台持续轮询：
    │     ├── 气压传感器：低于阈值 → emit sAlarmTurnOffSig()
    │     │     → ZKSort::alarmTurnOffFeedSlt() → 停止供料+关阀
    │     ├── 料位传感器：料斗空了 → 报警
    │     ├── 温度传感器：过热 → 报警
    │     └── 定时清灰：每隔 N 分钟 → emit sWipeStartSig()
    │           → ZKSort::myWipeStartSlt() → 擦拭相机镜头
    │
    └── 统计数据实时累计：
          struGsh.struStatistics[...].nThroughout（产量计数）
          struGsh.chanelArithEjectTimes[...]（每个通道的吹气次数）
            ↓ 定时（如每 30 分钟）
          MySqlite::updateStagedThroughout() → updateTable()
            ↓
          /sdcard/ts/statisticDataLog.db（持久化到 SQLite）
```

### 阶段 4：用户停止分选

```
用户在主页面点 [停止]
    │
    ▼
NewMainPage::on_runBtn_clicked()
  └── myFlow.onOff()
        ├── 停止供料：myFlow.resetFeederRG() → 振动器停
        ├── 关闭光源：myFlow.resetLampPlcOff()
        ├── 停止 IPC 检测：myFlow.configIpcSortStart(0)
        └── 关闭喷阀：myFlow.vavleOnOff() → 不再喷气
```

### 阶段 5：工程师调参（在运行中或停止时）

```
用户进入设置 → 灵敏度调整:
  SenseSetWidget::onSenseValueChanged(ARITH_GREY_A)
    → struCnfp.struGroupIdentify[...].struGreyColor[0].nSensLow = 55
    → myFlow.materialIdentifyParamsSend(ARITH_GREY_A, ...)
      → GlobalFlow 组包 → myQueue.push → 串口 → FPGA
      → FPGA 实时更新阈值 → 分选结果立即变化

用户进入设置 → 方案管理 → 切换方案:
  ModeManageWidget::apply()
    → myFlow.saveProfile()（保存当前方案到 JSON）
    → myFlow.getProfile()（加载新方案从 JSON）
    → 所有参数重新下发到硬件
    → 分选行为按新方案执行

用户进入工程师调试 → 图像采集:
  ImageCaptureWidget → TCP 接收图像 → ImageWidget 显示
    → 工程师观察好米/坏米的图像特征
    → 在图像上标记好料/坏料像素
    → SvmImageWidget → 训练 SVM 模型
    → 新模型系数写入 struGsh.m_svmcoefs[]
    → 下发到 FPGA → 改进分选效果
```

### 阶段 6：关机

```
用户在主页面点 [关机]

  ├── 如果正在分选 → 先停止分选

  └── system("shutdown -h now") → Linux 关机
```

## 五、协议支持体系 — 各通道生效位置

```
                    六条通信通道

┌─────────────────────────────────────────────────────────────────┐
│ 串口 1 (/dev/ttyS1) — 主通信通道                               │
│   协议: 自定义二进制帧 [0xAA 0x55][板类型][板编号][CMD][LEN]    │
│         [DATA...][校验]                                         │
│   指令集: 70+ 条 CMD_xxx（CMD_CAMERA_ANA_GAIN, CMD_ALL_SORT...）│
│   通信对象: FPGA主板、相机板、接口板                             │
│   生效位置: GlobalFlow::reset*() 所有函数                        │
│   数据量: 小（参数/指令，几十到几百字节）                        │
├─────────────────────────────────────────────────────────────────┤
│ 串口 2 (/dev/ttyS2) — 辅助控制通道                              │
│   协议: 同上格式，不同板类型                                     │
│   通信对象: 控制板（供料/报警/履带）、恒流源板（灯光）、PLC     │
│   生效位置: GlobalFlow::resetFeederRG(), resetLampRG(),         │
│             PlcAutoCtrlManager                                   │
│   数据量: 小                                                    │
├─────────────────────────────────────────────────────────────────┤
│ TCP (mynetwork) — 图像/波形通道                                 │
│   协议: 自定义 TCP 流，含 BMP 文件头                             │
│   通信对象: 相机板                                               │
│   生效位置: MyNetWorkThread（接收线程）                          │
│   数据量: 大（单帧图像可达几百 KB）                              │
├─────────────────────────────────────────────────────────────────┤
│ UDP (myudpthread) — IPC 相机控制通道                            │
│   协议: 自定义 UDP 报文，指令码 0x0001~0x5101                    │
│   通信对象: IPC 智能相机（每台一个 Linux 工控机）                │
│   生效位置: upgrademodelwidget, configIpcClassParams,           │

│   数据量: 中等（AI 模型文件可达几十 MB）                         │
├─────────────────────────────────────────────────────────────────┤
│ MQTT (mqttsrv) — 云平台通道                                     │
```

## 七、关键数据流总结

```
用户操作 → UI 控件
    ↓
修改全局结构体 (struCnfg / struCnfp / struGsh)
    ↓
调用 myFlow → GlobalFlow 组包
    ↓
myQueue.push() → 环形缓冲区
    ↓
mySerialSendThread → write(串口) → FPGA/控制板/接口板
    ↓
硬件执行 → 回包
    ↓
mySerialRecvThread → read(串口) → 解析 → 更新 struGsh
    ↓
后台线程检测变化 → emit 信号
    ↓
ZKSort 槽函数 → 自动响应（报警、清灰、关机...）

保存配置（显式或关机时）:
struCnfg / struCnfp → JsonDataConvert → CMyJson → JSON 文件 → SD 卡

统计持久化（定时自动）:
struGsh.struStatistics → MySqlite → SQLite .db 文件
```
