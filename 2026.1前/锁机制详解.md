# 锁（Mutex）— 多线程的数据保护

## 它解决的问题

两个线程同时写同一个变量：

```cpp
int count = 0;

// 线程 A                       // 线程 B
void addA() {                  void addB() {
    int tmp = count;  // 读到 0      int tmp = count;  // 也读到 0
    tmp = tmp + 1;    // 算成 1      tmp = tmp + 1;    // 也算成 1
    count = tmp;      // 写入 1      count = tmp;      // 写入 1
}                                  }
```

**两个线程各加了一次，count 应该是 2，结果却是 1。** 因为读和写之间没有保护，B 在 A 写完之前就读了旧值。这叫做**竞态条件**（race condition）。

---

## 锁怎么解决

```cpp
int count = 0;
QMutex m_lock;

// 线程 A                       // 线程 B
void addA() {                  void addB() {
    m_lock.lock();                m_lock.lock();    // ← 排队等
    int tmp = count;              // B 卡在这里
    tmp = tmp + 1;
    count = tmp;
    m_lock.unlock();              // A 放锁后 B 拿到锁
}                                  int tmp = count;  // 读到 1
                                   tmp = tmp + 1;    // 算成 2
                                   count = tmp;      // 写入 2 ✓
                                   m_lock.unlock();
                               }
```

## 锁的本质

```
锁 = 一扇只能通过一个人的门

线程 A 拿到锁 → 进去 → 门锁上
线程 B 想进去 → 推门 → 锁着 → 站门口等
线程 A 出来 → 开门 → 线程 B 立刻进去 → 门又锁上
```

**同一时刻只有一个线程能持有锁。** 其他想拿锁的线程必须排队等。

---

## 底层实现

锁的底层依赖于 CPU 提供的**原子操作指令**（如 x86 的 `CMPXCHG`、ARM 的 `LDREX/STREX`）：

```cpp
// QMutex::lock() 简化原理
void lock() {
    while (true) {
        // CPU 原子操作：如果锁是 0，设为 1，返回旧值
        // 整个过程不可分割，不会出现读到一半被别的线程打断
        if (atomic_exchange(&flag, 0, 1) == 0) {
            return;  // 拿到锁了
        }
        // 没拿到 → 让出 CPU 短暂休眠 → 再试
        usleep(100);
    }
}
```

## C++ 中的三种锁

| 类型 | 机制 | 适用场景 |
|------|------|---------|
| `std::mutex` / `QMutex` | 互斥锁，同一时刻只有一个线程持有 | 保护不允许多线程同时访问的数据 |
| `std::shared_mutex` / `QReadWriteLock` | 读写锁，多线程可同时读，写时独占 | 读多写少的场景 |
| `std::atomic` | 无锁原子操作，CPU 指令级别保证 | 单个变量（int/bool）的简单读写 |

## 在这个项目中的使用

```cpp
// myQueue — 串口数据队列，多线程共用
class MySerialQueue {
    QMutex m_queueLock;    // 保护 m_queue 的并发访问

    bool push(char* data) {
        QMutexLocker locker(&m_queueLock);
        // ↑ 上锁
        // 操作共享数据 m_queue（rear 指针、data 数组）
        return _push(data);
    }   // ↓ locker 析构 → 解锁
};
```

```
GlobalFlow (主线程)            mySerialSendThread (发送线程)
        │                              │
        ├─ myQueue.push(A) ──→         ├─ myQueue.pop() → write 串口
        ├─ myQueue.push(B)             ├─ myQueue.pop() → write 串口
        └─ myQueue.push(C)             └─ ...
         
每次 push/pop 都要先拿锁 → 两个线程永远不会同时碰 m_queue 的内部数据 → 安全
```

## QMutexLocker — 自动锁管理

手动 lock/unlock 的问题是函数可能有多个出口，任何一个忘了 unlock 就会死锁：

```cpp
void doSomething()
{
    m_lock.lock();
    if (someError) {
        return;            // ← 忘了 unlock！→ 死锁
    }
    m_lock.unlock();
}
```

QMutexLocker 用 RAII 解决：

```cpp
void doSomething()
{
    QMutexLocker locker(&m_lock);   // 构造时自动 lock()
    if (someError) {
        return;             // locker 析构时自动 unlock() ✓
    }
}   // 无论从哪出去，locker 析构 → 自动 unlock()
```

## 锁的代价

| 代价 | 说明 |
|------|------|
| 性能 | 等锁时线程挂起，切换开销 |
| 死锁 | A 等 B 释放锁，B 等 A 释放锁 → 互相卡死 |
| 过度保护 | 锁范围太大 → 串行化 → 多线程白开了 |

**原则：锁的范围越小越好，只保护真正共享的数据。**

---

## 一句话

锁 = 多线程世界里的"红绿灯"。没有红绿灯，十字路口会撞车（数据错乱）；有了红绿灯，一次只过一个方向（线程安全），但要排队等（性能代价）。QMutexLocker 就是自动帮你关红绿灯的工具——上了路就锁，下了路就开，不会忘。
