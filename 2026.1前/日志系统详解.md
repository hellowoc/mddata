# main() 日志系统详解

---

## 总览

```
main() 中的日志代码分为两大段：

段落一：日志文件管理（第108行 ~ 第168行）
        └── 纯手工操作：数量控制、跨天迁移、日期解析
            不依赖任何日志框架

段落二：产品UI日志  （第170行 ~ 第194行）
        └── Log4Qt 框架初始化：Properties 配置 → PropertyConfigurator → Logger
            应用层后续代码通过 logger()->info(...) 写日志
```

两段是**互补关系**：段落一做的是框架管不到的事（启动时清理旧日志），段落二才是真正的运行时日志框架。

## 段落一：日志文件管理（手工操作）

### 代码原文

```cpp
//日志文件管理
#ifdef Q_OS_WIN
    QString path = qApp->applicationDirPath()+"/logs/";
#else
    QString path = "/sdcard/logs/";
#endif

QDir dir(path);
QStringList filter;
filter<<"log.txt.*";
dir.setNameFilters(filter);
QList<QFileInfo> fileInfo = QList<QFileInfo>(
    dir.entryInfoList(filter, QDir::AllEntries, QDir::Name));

if(fileInfo.size() >= 30){
    QFile::remove(fileInfo.at(0).absoluteFilePath());
}

QString dateString = QDateTime::currentDateTime().toString("yyyy-MM-dd");
QString name = "log.txt";
#ifdef Q_OS_WIN
    QFileInfo logFileInfo(qApp->applicationDirPath()+"/logs/"+name);
#else
    QFileInfo logFileInfo("/sdcard/logs/"+name);
#endif

if (logFileInfo.exists() && logFileInfo.size() > 0)
{
    QStringList sections ;
    QFile file(logFileInfo.absoluteFilePath());
    if(file.open(QIODevice::ReadOnly | QIODevice::Text))
    {
        while (!file.atEnd())
        {
            QByteArray line = file.readLine();
            QString str(line);
            if(str == "header" || str == "footer") continue;
            sections =  str.split(QRegExp(" "));
            if(sections.at(0).contains("-"))
            {
                break;
            }
        }
    }
    file.close();
    if (sections.at(0) != "")
    {
        if (sections.at(0) != "")
        {
            if (sections.at(0) !=  dateString)
            {
                g_Runtime().copyFileToPath(
                    logFileInfo.absoluteFilePath(),
                    logFileInfo.absoluteFilePath()+"."+sections.at(0),
                    true);
                g_Runtime().delFile(logFileInfo.absoluteFilePath());
            }
        }
    }
}
```

### 子段落 A：日志数量上限控制

```cpp
#ifdef Q_OS_WIN
    QString path = qApp->applicationDirPath()+"/logs/";
#else
    QString path = "/sdcard/logs/";
#endif
```

**平台路径切换：**

| 宏条件 | 生效平台 | 日志目录 | 含义 |
|--------|---------|---------|------|
| `Q_OS_WIN` 定义 | Windows（开发调试用） | `程序所在目录/logs/` | 如 `D:\project\zksort\logs\` |
| `Q_OS_WIN` 未定义 | Linux ARM（实际产品） | `/sdcard/logs/` | 设备内部存储的固定路径 |

`sdcard` 是嵌入式 Linux 上 SD 卡/TF 卡的挂载点，不一定是真正的 SD 卡，也可能是 eMMC 芯片。这是一个行业惯用命名。

---

#### QDir — 目录对象

```cpp
QDir dir(path);
```

```
QDir
  继承: 无（独立工具类，不继承 QObject）
  定位: Qt 中操作目录的类

dir(path) 构造时做了什么？
  → 存储路径字符串，不做任何磁盘 I/O
  → 后续调用 entryInfoList() 时才真正访问文件系统
```

QDir 是目录的"入口"。有了它才能列举目录下的文件、创建/删除目录、设置过滤条件。

---

#### QStringList filter — 文件名过滤器

```cpp
QStringList filter;
filter << "log.txt.*";
```

```
QStringList
  继承: QList<QString>
  实质: 字符串数组

filter 当前内容: ["log.txt.*"]
```

`"log.txt.*"` 是文件名的通配符模式，只匹配 `log.txt.2026-05-19` 这样的归档日志文件，**不匹配** 正在写的 `log.txt`。

---

#### setNameFilters + entryInfoList — 列举并排序

```cpp
dir.setNameFilters(filter);
QList<QFileInfo> fileInfo = QList<QFileInfo>(
    dir.entryInfoList(filter, QDir::AllEntries, QDir::Name));
```

这一行做了三件事：

##### 1. `dir.setNameFilters(filter)`

```
setNameFilters(["log.txt.*"])
  → 告诉 QDir: 后续列举文件时，只保留名字匹配 "log.txt.*" 的
  → 纯内存操作，不访问磁盘
```

##### 2. `dir.entryInfoList(filter, QDir::AllEntries, QDir::Name)`

```
entryInfoList(filter, QDir::AllEntries, QDir::Name)
  → 系统调用: ::opendir("/sdcard/logs")
              ::readdir() 循环读取每个目录项
              ::stat() 获取每个文件的元信息
              ::closedir()
  → 只返回匹配 filter 的文件
  → 按 QDir::Name 排序（按文件名字典序）
```

三个参数的含义：

| 参数 | 值 | 含义 |
|------|-----|------|
| `filter` | `["log.txt.*"]` | 文件名过滤规则 |
| `QDir::AllEntries` | 枚举值 | 列出文件和目录（不跳过任何类型） |
| `QDir::Name` | 排序方式 | 按文件名字典序排列 |

**为什么用 `QDir::Name` 排序？**

归档文件名格式是 `log.txt.2026-05-19`、`log.txt.2026-05-20`... 字典序排列后，最老的日期排在最前面。所以 `fileInfo.at(0)` 一定是最老的文件。

##### 3. `QList<QFileInfo>(...)` — 类型转换

`entryInfoList` 返回 `QFileInfoList`（即 `QList<QFileInfo>`），这里显式构造了一次，实际上可以省略，写出来是为了更清晰地表明类型。

---

#### 删除最老的文件

```cpp
if (fileInfo.size() >= 30) {
    QFile::remove(fileInfo.at(0).absoluteFilePath());
}
```

```
fileInfo.size() >= 30
  → 归档文件数 ≥ 30 个时才触发删除

fileInfo.at(0)
  → 取第一个（最老的，因为已按 Name 排序）

fileInfo.at(0).absoluteFilePath()
  → 返回如 "/sdcard/logs/log.txt.2026-04-20" 的绝对路径

QFile::remove(...)
  → 这是 QFile 的静态方法
  → 系统调用: ::unlink("/sdcard/logs/log.txt.2026-04-20")
  → 静态方法 = 不需要先构造 QFile 对象，直接调
```

**为什么是 30 个？**

每个日志文件约 1MB，30 个 = 30MB = 大约一个月的历史。超过一个月的 bug 通常不会再被追溯。如果空间非常紧张可以改小（如 10 个），如果合规要求长期保留则改大（如 365 个）。

### 子段落 C：跨天日志迁移（核心逻辑）

这是整段代码中最复杂的部分。

#### 前置判断

```cpp
if (logFileInfo.exists() && logFileInfo.size() > 0)
```

```
logFileInfo.exists()
  → ::stat("/sdcard/logs/log.txt")  ← 检查当前日志文件是否存在

logFileInfo.size() > 0
  → 检查文件大小 > 0，避免空文件被误迁移
```

两个条件都满足（log.txt 存在且有内容）才进入迁移逻辑。第一次开机时 log.txt 不存在，直接跳过。

---

#### 打开文件并逐行读取

```cpp
QStringList sections;
QFile file(logFileInfo.absoluteFilePath());
```

##### QFile — 文件 I/O 对象

```
QFile
  继承: QFileDevice → QIODevice → QObject
  定位: Qt 中读写文件的类

file(logFileInfo.absoluteFilePath())
  构造: 只存路径，不打开文件
```

QFile 和 QFileInfo 的区别：

| 类 | 用途 | 类比 |
|----|------|------|
| `QFileInfo` | 查询文件元信息（大小、是否存在、路径） | 查档案目录 |
| `QFile` | 打开并读写文件内容 | 取出档案翻看 |

---

##### file.open() — 打开文件

```cpp
if (file.open(QIODevice::ReadOnly | QIODevice::Text))
```

```
file.open(QIODevice::ReadOnly | QIODevice::Text)
  → 系统调用: ::open("/sdcard/logs/log.txt", O_RDONLY)

两个标志:
  QIODevice::ReadOnly   = 只读模式（不写）
  QIODevice::Text       = 文本模式
    → 自动处理不同平台的换行符（\r\n → \n）
    → 不影响 Linux，但保证跨平台行为一致

成功 → 返回 true → 进入 while 循环
失败 → 返回 false → 跳过整个迁移逻辑
```

---

##### while 循环 — 逐行读取

```cpp
while (!file.atEnd())
{
    QByteArray line = file.readLine();
    QString str(line);
    if (str == "header" || str == "footer") continue;
    sections = str.split(QRegExp(" "));
    if (sections.at(0).contains("-"))
    {
        break;
    }
}
```

**`file.atEnd()`** — 检查是否读到文件末尾：

```
file.atEnd()
  → 返回 true = 读完了，false = 还有内容
  → Qt 内部维护一个位置指针，readLine 后自动推进
```

**`file.readLine()`** — 读一行：

```
file.readLine()
  → 系统调用: ::read(fd, buf, ...) 读入内部缓冲区
  → 返回 QByteArray（原始字节数组，不是 QString）
  → 包含换行符 "\n"
```

**`QString str(line)`** — 字节转字符串：

```
QString str(line)
  → QByteArray → QString 转换
  → 使用当前设置的编码（main 前面设置过 UTF-8）
```

**跳过 header/footer 行：**

```cpp
if (str == "header" || str == "footer") continue;
```

Log4Qt 的 PatternLayout 在日志文件开头写入 header 字符串（本项目配置为"程序启动"），结尾写入 footer 字符串（"程序结束"）。这两行不是真正的日志数据，需要跳过。

**按空格分割：**

```cpp
sections = str.split(QRegExp(" "));
```

```
str.split(QRegExp(" "))
  → 以空格为分隔符把字符串切成多段

示例输入:
  "2026-05-19 14:30:00 [Mainthread] INFO ZKSort - power on"

sections 结果:
  [0] "2026-05-19"
  [1] "14:30:00"
  [2] "[Mainthread]"
  [3] "INFO"
  [4] "ZKSort"
  ... 等
```

**QRegExp** — Qt 的正则表达式类（Qt4 时代）。这里只用了一个最简单的模式（匹配单个空格），相当于 `QString::split(' ')`，但用 `QRegExp` 是为了兼容写法。

**检查日期格式：**

```cpp
if (sections.at(0).contains("-"))
{
    break;
}
```

`sections.at(0)` 是第一段。如果包含 `-`，说明是日期格式（如 `2026-05-19`），说明找到了第一条有效的日志记录。`break` 跳出循环，取这行的日期作为文件中的上一条日志日期。

**为什么只读第一条日志？**

不需要遍历整个文件。只需要知道文件中最早的日志是哪天写的，然后判断是否跨天。而日志是按时间顺序追加的，所以第一行日志就是最早的日期。

---

##### file.close() — 关闭文件

```cpp
file.close();
```

```
file.close()
  → 系统调用: ::close(fd)
  → 释放文件描述符
```

不关也行（QFile 析构时会自动关），但显式关闭是好习惯——尽早释放资源。

---

#### 跨天判断与迁移执行

```cpp
if (sections.at(0) != "")
{
    if (sections.at(0) != "")           // 冗余判断（原代码如此）
    {
        if (sections.at(0) != dateString)
        {
            g_Runtime().copyFileToPath(
                logFileInfo.absoluteFilePath(),
                logFileInfo.absoluteFilePath()+"."+sections.at(0),
                true);
            g_Runtime().delFile(logFileInfo.absoluteFilePath());
        }
    }
}
```

**第一个 `if (sections.at(0) != "")`：** 确保读到了有效日志行（如果文件全是 header/footer 行，sections 为空，直接跳过）。

**冗余判断：** 内层的第二个 `if (sections.at(0) != "")` 和第一个完全一样——原代码的冗余，不影响逻辑。

**核心判断：`sections.at(0) != dateString`**

```
sections.at(0)     = "2026-05-19"  (日志文件中第一条记录的日期)
dateString         = "2026-05-21"  (今天的日期)

不等 → 跨天了 → 需要迁移
相等 → 同一天 → 不需要迁移（可能是同天重启）
```

**迁移操作一：拷贝归档**

```cpp
g_Runtime().copyFileToPath(
    logFileInfo.absoluteFilePath(),                   // 源: /sdcard/logs/log.txt
    logFileInfo.absoluteFilePath()+"."+sections.at(0),// 目标: /sdcard/logs/log.txt.2026-05-19
    true);                                            // 覆盖已存在的
```

```
源: /sdcard/logs/log.txt
目标: /sdcard/logs/log.txt.2026-05-19
                         ^^^^^^^^^^
                         从日志内容中提取的日期（不是今天的日期）
```

第三个参数 `true` = 如果目标文件已存在则覆盖，防止因异常重启导致的重复归档。

**迁移操作二：删除原文件**

```cpp
g_Runtime().delFile(logFileInfo.absoluteFilePath());
```

```
delFile("/sdcard/logs/log.txt")
  → QFile::remove()
    → ::unlink("/sdcard/logs/log.txt")
```

为什么要先拷贝再删除？因为 `rename()` 如果目标已存在会覆盖，但跨文件系统 rename 可能失败。先拷贝再删除是最安全的方式。

**迁移的效果：**

```
迁移前:
  /sdcard/logs/log.txt          ← 内容是昨天的日志

迁移后:
  /sdcard/logs/log.txt.2026-05-19  ← 昨天的日志归档
  /sdcard/logs/log.txt             ← 不存在了（等 Log4Qt 启动后新建）
```

## 段落二：产品UI日志（Log4Qt 框架初始化）

### 代码原文

```cpp
//产品UI日志
Log4Qt::Properties qtlogProper;

qtlogProper.setProperty("log4j.rootLogger", "DEBUG, A1");

qtlogProper.setProperty("log4j.appender.A1",
    "org.apache.log4j.DailyRollingFileAppender");
qtlogProper.setProperty("log4j.appender.A1.appendFile", "true");
qtlogProper.setProperty("log4j.appender.A1.immediateFlush", "true");
qtlogProper.setProperty("log4j.appender.A1.datePattern", "'.'yyyy-MM-dd");
qtlogProper.setProperty("log4j.appender.A1.file",
    logFileInfo.absoluteFilePath());

qtlogProper.setProperty("log4j.appender.A1.layout",
    "org.apache.log4j.PatternLayout");
qtlogProper.setProperty("log4j.appender.A1.layout.ConversionPattern",
    "%d [%t] %p %c%x - %m%n");
qtlogProper.setProperty("log4j.appender.A1.layout.header", "程序启动");
qtlogProper.setProperty("log4j.appender.A1.layout.footer", "程序结束");

Log4Qt::PropertyConfigurator::configure(qtlogProper);
Log4Qt::Logger* logger = Log4Qt::Logger::rootLogger();
```

这段代码的本质是用 **C++ 代码模拟 Java 的 `log4j.properties` 文件**。实际上完全等价于写一个 properties 文件后加载，但用代码方式可以在运行时动态决定某些值（如文件路径）。

---

### Log4Qt 的架构

Log4Qt 完全模仿了 Apache log4j 的设计，三个核心概念：

```
Logger           → "谁在写日志"   (ZKSort, FPGA, GlobalFlow...)
    ↓
Appender         → "日志写到哪"   (DailyRollingFileAppender → /sdcard/logs/log.txt)
    ↓
Layout           → "写成什么样"   (PatternLayout → 2026-05-21 14:30 [Mainthread] INFO ...)
```

一条日志消息的流程：

```
logger()->info("power on: %1", cnt)
    ↓
Logger 判断: 级别（INFO）>= 阈值（DEBUG）? → 是，继续
    ↓
Appender 接收: DailyRollingFileAppender
    ↓
Layout 格式化: PatternLayout 应用转换模式
    ↓
写入磁盘: /sdcard/logs/log.txt
```

### Log4Qt::Properties — 配置键值对

```cpp
Log4Qt::Properties qtlogProper;
```

```
Log4Qt::Properties
  继承: QHash<QString, QString>
  实质: 一个 key-value 映射表，key 和 value 都是字符串
```

### 逐条配置项拆解

#### rootLogger 配置

```cpp
qtlogProper.setProperty("log4j.rootLogger", "DEBUG, A1");
```

```
格式: log4j.rootLogger = <日志级别>, <使用的Appender名称>

```

---

#### Appender 定义

```cpp
// 指定 A1 是 DailyRollingFileAppender
qtlogProper.setProperty("log4j.appender.A1",
    "org.apache.log4j.DailyRollingFileAppender");

// 追加模式（不覆盖已有内容）
qtlogProper.setProperty("log4j.appender.A1.appendFile", "true");

// 每条日志立即 flush（断电尽可能不丢日志）
qtlogProper.setProperty("log4j.appender.A1.immediateFlush", "true");

// 按天滚动的日期模式
qtlogProper.setProperty("log4j.appender.A1.datePattern", "'.'yyyy-MM-dd");

// 日志文件路径
qtlogProper.setProperty("log4j.appender.A1.file",
    logFileInfo.absoluteFilePath());
```

**DailyRollingFileAppender** — 按时间自动滚动文件：

```
DailyRollingFileAppender
  继承: FileAppender → WriterAppender → AppenderSkeleton → Appender → QObject

工作机制:
  → 构造时读取 datePattern 和当前时间，计算下次滚动的时刻
  → 每次写日志前检查: 是否到了滚动时刻？
      是 → 将当前 log.txt 重命名为 log.txt.2026-05-21
           → 重新创建新的 log.txt 继续写
      否 → 继续写

datePattern = "'.'yyyy-MM-dd"
  → '.' 是字面量（文件名中真的出现一个点）
  → yyyy-MM-dd 按当前日期格式化
  → 结果: 文件名后缀为 .2026-05-21
  → 滚动时刻: 每天午夜 00:00:00

其他可选模式:
  "'.'yyyy-MM-dd-hh"    每分钟滚动
  "'.'yyyy-MM-dd-hh-mm" 每小时滚动
  "'.'yyyy-ww"          每周滚动
```

**`appendFile = "true"`**：

```
true  = 追加模式。如果 log.txt 已存在，新日志追加到末尾
false = 覆盖模式。每次启动清空旧日志从头写

对于工业设备，true 是必要的——不希望重启丢失上一次运行的日志
```

**`immediateFlush = "true"`**：

```
true  = 每条日志消息后立即 fflush() → 调用 ::fsync()
        → 日志几乎实时写入磁盘
false = 写入内部缓冲区，攒够一批再写

对工业设备选 true 的原因:
  如果断电 → false 可能丢几秒到几分钟的日志
             → 崩溃发生时最近几条日志是排查问题最关键的
```

代价是对 Flash 的擦写更多。可选的折中方案是 `false` + 每 5 秒定时 sync。

#### Layout 定义

```cpp
qtlogProper.setProperty("log4j.appender.A1.layout",
    "org.apache.log4j.PatternLayout");

qtlogProper.setProperty("log4j.appender.A1.layout.ConversionPattern",
    "%d [%t] %p %c%x - %m%n");

qtlogProper.setProperty("log4j.appender.A1.layout.header", "程序启动");
qtlogProper.setProperty("log4j.appender.A1.layout.footer", "程序结束");
```

**PatternLayout** — 用格式字符串定义每条日志的外观：

```
PatternLayout
  继承: Layout → LogObject → QObject

需要配置:
  conversionPattern = 每条日志消息的格式
  header            = 日志文件开头写入一次
  footer            = Appender 关闭时写入文件末尾
```

**ConversionPattern 详解：**

```
"%d [%t] %p %c%x - %m%n"
  │    │    │   ││  │  └── %n : 换行符
  │    │    │   ││  └────── %m : 实际日志消息内容
  │    │    │   │└───────── %x : NDC（嵌套诊断上下文，不常用）
  │    │    │   └────────── %c : Logger 名称（类名）
  │    │    └────────────── %p : 日志级别（DEBUG/INFO/WARN/ERROR/FATAL）
  │    └─────────────────── %t : 线程名称
  └──────────────────────── %d : 日期时间
```

**实际输出效果：**

```
2026-05-21 14:30:00.123 [Mainthread] INFO  ZKSort       - power on: 1523 times
2026-05-21 14:30:05.456 [mySerialRecvThread] WARN  myQextSerialPort - receive timeout
2026-05-21 14:35:12.789 [updateStatusThread] ERROR fpga          - write reg failed: 0x1A3F
                                ^^^^^^                              ^^^^^^^^^^^^^^^^
                                %p                                  %m
```

**header / footer：**

```
header = "程序启动"
  → Appender 激活时写入文件第一行
  → 标记一次程序运行的开始

footer = "程序结束"
  → Appender 关闭时写入文件最后一行
  → 标记一次程序运行的结束

这两个就是段落一中 while 循环里被跳过的行：
  if(str == "header" || str == "footer") continue;
```

#### PropertyConfigurator — 把配置"激活"

```cpp
Log4Qt::PropertyConfigurator::configure(qtlogProper);
```

```
PropertyConfigurator::configure(qtlogProper)  ← 静态方法

内部做的事:
  1. 解析 qtlogProper 中的所有键值对
  2. 识别 "log4j.rootLogger" → 设置根 Logger 的级别为 DEBUG，关联 Appender "A1"
  3. 识别 "log4j.appender.A1" → 创建 DailyRollingFileAppender 实例
  4. 识别 "log4j.appender.A1.*" → 设置 Appender 的各项属性
  5. 识别 "log4j.appender.A1.layout" → 创建 PatternLayout 实例
  6. 识别 "log4j.appender.A1.layout.*" → 设置 Layout 的属性
  7. 将 Appender 注册到 LoggerRepository（全局日志仓库）

执行完这一行后，Log4Qt 框架完全就绪。
```

这是一个**静态方法**，不需要创建 PropertyConfigurator 实例，直接 `ClassName::staticMethod()` 调用。

#### rootLogger — 获取根日志器

```cpp
Log4Qt::Logger* logger = Log4Qt::Logger::rootLogger();
```

```
Log4Qt::Logger::rootLogger()
  → 返回指向全局根 Logger 的指针
  → 这是一个单例（整个程序只有一个 root logger）
  → 存放在 LoggerRepository 中

logger 指针在 main() 最后用于清理:
  logger->removeAllAppenders();      // 移除所有 Appender
  logger->loggerRepository()->shutdown(); // 关闭日志系统
```

**这个 logger 指针在 main() 中的使用很有限。** 实际的日志写入是通过各个类的 `logger()` 成员函数完成的（由 `LOG4QT_DECLARE_QCLASS_LOGGER` 宏提供）：

```cpp
// zksort.h 中
class ZKSort : public BaseUi
{
    Q_OBJECT
    LOG4QT_DECLARE_QCLASS_LOGGER   // ← 这个宏给 ZKSort 类加了 logger() 成员
    ...
};

// zksort.cpp 中
ZKSort::ZKSort(...) {
    logger()->info("power on: %1 times", onoff_cnts);
    //         ^^
    // 这个 logger() 返回的是 rootLogger 的一个子 Logger
    // 自动以类名 "ZKSort" 命名
}
```

`LOG4QT_DECLARE_QCLASS_LOGGER` 宏展开后：

```cpp
// 展开后的等价代码：
private:
    mutable Log4Qt::ClassLogger mLog4QtClassLogger;
public:
    inline Log4Qt::Logger *logger() const {
        return mLog4QtClassLogger.logger(this);
    }
```

`ClassLogger` 内部会：
1. 获取 this 指针指向的对象的类名（通过 Qt 元对象系统）
2. 以类名为名称，向 LoggerRepository 请求一个 Logger
3. 返回这个 Logger 指针

所以每个类自动拥有一个以自己类名命名的 Logger，日志输出中 `%c` 部分就会显示类名。

### 程序退出时的清理

```cpp
int execResult = a.exec();

if (splash) {
    splash->finish(&w);
    delete splash;
}
logger->removeAllAppenders();             // 移除所有 Appender
logger->loggerRepository()->shutdown();   // 关闭日志仓库
return execResult;
```

**`removeAllAppenders()`**：

移除 rootLogger 上关联的所有 Appender。Appender 析构时会调用 `close()`，DailyRollingFileAppender 的 `close()` 会把文件描述符关闭。

**`loggerRepository()->shutdown()`**：

关闭整个日志仓库，释放所有资源。这个调用不是必须的（程序马上就要退出了），但它是 Log4Qt 的正式清理流程，加上更规范。
