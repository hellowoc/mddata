# 系统时序与进程/线程分析

---

## 一、从开机到关机的完整时序

### 阶段 1：main() 启动 — 单线程（Mainthread）

```
进程: ./zksort (唯一进程)
线程: 1 个 — Mainthread

时间线:
  T=0ms    main() 入口
  T=1ms    zkApplication 构造（Qt 初始化）
  T=2ms    UTF-8 编码设置
  T=3ms    USB 触发检查（如果 U 盘插入）
  T=50ms   启动画面显示（boot.bmp）
  T=51ms   日志管理（清理旧文件、跨天迁移）
  T=52ms   Log4Qt 配置
  T=53ms   资源包注册（resource.rcc）
  T=54ms   GlobalFlow::initAll()  ← ★ 线程爆炸点
```

### 阶段 2：initAll() — 线程批量创建

```
GlobalFlow::initAll() 内部（全部在主线程中执行）:

  T=55ms   读 JSON 配置文件 → 填充 struCnfg / struCnfp
  T=60ms   initGsh() 初始化共享状态
  T=65ms   initIpcAttr() 初始化 IPC 属性
  T=70ms   打开串口 1: mySerial.comOpen("/dev/ttyS1")
  T=71ms   打开串口 2: mySerial.comOpen("/dev/ttyS2")

  T=72ms   创建发送线程 1 ★  mySerialSend = new mySerialSendThread;   start()
  T=73ms   创建接收线程 1 ★  mySerialRecv = new mySerialRecvThread;   start()
  T=74ms   创建发送线程 2 ★  mySerial2Send = new mySerial2SendThread;  start()
  T=75ms   创建接收线程 2 ★  mySerial2Recv = new mySerial2RecvThread;  start()

  T=76ms   创建状态轮询线程 ★  myUpdateStatusThread = new updateStatusThread;
                              start()
  T=80ms   创建网络线程（如果需要）
  T=85ms   创建 MQTT 线程（仅 Linux）
```

**此时系统中同时存在的线程：**

```
┌─────────────────────────────────────────────────────┐
│ Mainthread          主线程（事件循环）               │
│ mySerialSendThread   串口 1 发送                     │
│ mySerialRecvThread   串口 1 接收                     │
│ mySerial2SendThread  串口 2 发送                     │
│ mySerial2RecvThread  串口 2 接收                     │
│ myUpdateStatusThread 状态轮询（已创建，首动作 sleep） │
│ [myNetWorkThread]    TCP 图像接收（如果配置了网络）    │
│ [mqttThread]         MQTT 长连接（仅 Linux）          │
│ [mqttMsgParaseThread] MQTT 消息解析（仅 Linux）       │
└─────────────────────────────────────────────────────┘
              共: 6-9 个线程
```

### 阶段 3：回到 main() — 完成初始化

```
  T=90ms   语言加载（g_myLan().ChangeLanguage()）
  T=95ms   ZKSort 构造:
               setupUi → 创建 TopWidget, QStackedWidget, BottomWidget
               LoadSkin → 加载 QSS 样式
               g_MainManager().setStackedWidget()
               启动 mytimer（100ms 后触发）
  T=105ms  w.setFixedSize(1024, 768) + w.move(0,0) + w.show()
  T=110ms  a.exec() 进入事件循环
```

### 阶段 4：a.exec() — 事件循环启动后的第一个事件

```
  T=210ms  (100ms 定时器到期)
           mytimer → dccryptCheck()

  dccryptCheck() 内（仍在 Mainthread 中执行）:
    ├── 加密校验（读 EEPROM，解密，验证）
    ├── 通过 → communication()
    │         ├── 显示 "通信中，请稍候..."
    │         ├── myFlow.resetCommunication()
    │         │     └─ 通过 myQueue.push 发查询指令
    │         │        串口发送线程从队列取出 → write(串口fd) → 硬件
    │         │        串口接收线程 read(串口fd) → 收到回包 → 解析
    │         └── 成功 → startEnterMainPage()
    │               ├── myFlow.initSendAllParams()
    │               │     └─ 大量串口指令依次下发
    │               ├── 等待 IPC 模型加载（最多 15 秒）
    │               ├── CreateWidgets() → 创建主页
    │               └── ShowWidget(主页)
    │
    └── 末尾: connect × 15（连接所有后台信号到 ZKSort 槽函数）
        ★ 此时 updateStatusThread 的信号才真正接入系统
```

## 二、稳态运行时的线程全景

### 常驻线程（整个程序运行期间一直存在）

```
┌──────────────────────────────────────────────────────────────────┐
│ 1. Mainthread（主线程）                                           │
│    运行: a.exec() 事件循环                                        │
│    职责: 所有 UI 渲染、用户交互、页面切换                          │
│    不能做: 耗时操作（会卡 UI）、阻塞 I/O                           │
│    关键资源: QStackedWidget, 所有 UI 控件                          │
├──────────────────────────────────────────────────────────────────┤
│ 2. mySerialSendThread（串口 1 发送线程）                          │
│    运行: while(1) { myQueue.pop() → ::write(fd, data) }          │
│    职责: 从环形缓冲区取数据包，写入 /dev/ttyS1                     │
│    关键资源: myQueue（环形缓冲区共享）                              │
│    保护: QMutex（push/pop 互斥）                                   │
├──────────────────────────────────────────────────────────────────┤
│ 3. mySerialRecvThread（串口 1 接收线程）                          │
│    运行: while(1) { ::read(fd) → 解析协议 → 更新 struGsh }        │
│    职责: 从 /dev/ttyS1 读数据，解析帧头/命令码/校验，更新全局状态  │
│    信号: signalCtrlBoardOutage(), signalCtrlBoardFeedOnOffQuery() │
│          等 6 个信号 → PlcAutoCtrlManager 或 GlobalFlow 处理      │
├──────────────────────────────────────────────────────────────────┤
│ 4. mySerial2SendThread（串口 2 发送线程）                         │
│    运行: 同串口 1                                                 │
│    职责: 写入 /dev/ttyS2（控制板/灯板/PLC）                        │
│    关键资源: myQueue2                                             │
├──────────────────────────────────────────────────────────────────┤
│ 5. mySerial2RecvThread（串口 2 接收线程）                         │
│    运行: 同串口 1 接收                                            │
│    职责: 从 /dev/ttyS2 读数据                                     │
│    信号: → PlcAutoCtrlManager（PLC 指令处理）                      │
├──────────────────────────────────────────────────────────────────┤
│ 6. myUpdateStatusThread（状态轮询线程）                           │
│    运行: while(1) { sleep(100ms) → 查询硬件状态 → emit 信号 }     │
│    职责: 周期查询气压/料位/温度/清灰状态                           │
│    信号: sAlarmTurnOffSig(), sAlarmTurnOnSig(),                   │
│          sWipeStartSig(), sLostFpgaPowerOffSig(),                 │
│          sDccryptWarningSig(), ...（共 ~15 个信号）                │
│    接收者: ZKSort（主线程中的槽函数）                              │
├──────────────────────────────────────────────────────────────────┤
│ 7. myNetWorkThread / MyNetWorkThread（TCP 图像接收线程）          │
│    运行: while(1) { accept/read TCP → 组帧 → 通知 UI }           │
│    职责: 接收相机板的图像/波形数据（大块数据，KB~MB 级）           │
│    信号: 图像就绪 → ImageWidget 显示                               │
├──────────────────────────────────────────────────────────────────┤
│ 8. mqttThread（MQTT 长连接线程，仅 Linux）                         │
│    运行: while(1) { mqtt_loop() → 维持与 Broker 的连接 }          │
│    职责: 与云平台保持长连接，收发 MQTT 消息                         │
├──────────────────────────────────────────────────────────────────┤
│ 9. mqttMsgParaseThread（MQTT 消息解析线程，仅 Linux）              │
│    运行: while(1) { 解析 MQTT 消息 → 分派到对应处理逻辑 }         │
│    职责: 远程升级指令、模型下载、状态上报、控制指令                  │
│    信号: cmd_updatezksort_ask(), cmd_uploadIpcInfo_ask() 等       │
└──────────────────────────────────────────────────────────────────┘

              稳态常驻线程: 7-9 个
```

## 三、进程间关系

### 主进程启动的子进程

```
./zksort (主进程, PID=100)
    │
    ├── ./zktimer (守护进程)
    │     startDetached 方式启动
    │     职责: 监控系统时间，防止 RTC 被回拨（绕过加密）
    │     生命周期: 随主进程启动 → 随系统关机结束
    │     关系: 完全独立，不通信
    │
    ├── train.exe (SVM 训练)
    │     QProcess 方式启动
    │     职责: 外部 SVM 训练程序
    │     生命周期: 训练开始 → 完成 → 进程退出
    │     关系: 主进程等待其完成，读取输出模型文件
    │
    ├── /sdcard/wifi/wifiCtrol.sh (WiFi 控制)
    │     QProcess 方式启动
    │     职责: 扫描/连接/断开 WiFi
    │     生命周期: 每次 WiFi 操作 → 完成 → 进程退出
    │     关系: 主进程等待 stdout 输出，解析结果
    │
    ├── unzip (固件解压)
    │     QProcess 方式启动
    │     职责: 解压固件升级包
    │     生命周期: 升级开始 → 解压完成 → 进程退出
    │
    ├── arecord (喷阀测试录音)
    │     QProcess 方式启动
    │     职责: 录制喷阀动作声音用于判断喷阀是否正常
    │     生命周期: 快速测试期间
    │
    ├── findActiveIp.sh (网络扫描)
    │     QProcess 方式启动
    │     职责: 扫描局域网内的活跃 IP
    │     生命周期: 用户点"扫描" → 扫描完成 → 进程退出
    │
    └── /bin/sh -c "shutdown -h now" (关机)
          system() 方式调用
          生命周期: 关机时一次性
```

## 五、资源竞争与保护

### 5.1 共享数据结构及保护机制

```
struCnfg (全局配置):
  访问者: Mainthread（读/写）, updateStatusThread（读）
  保护: 无锁（读多写少，写操作都在 Mainthread 中完成）
  风险: 修改时如果 updateStatusThread 恰好读到 → 不一致（容忍）

struGsh (运行时状态):
  访问者: Mainthread（读/写）, mySerialRecvThread（写）, 
          updateStatusThread（读）
  保护: 无锁（volatile 变量 + 操作原子性假设）
  风险: 存在竞态条件，但嵌入式场景容忍度较高

myQueue / myQueue2 (串口环形缓冲区):
  访问者: GlobalFlow/Mainthread（push）, mySerialSendThread（pop）
  保护: QMutex m_queueLock（push/pop 都加锁）
  风险: 无

g_Runtime().m_sampleMap:
  访问者: Mainthread / SvmLearnThread（写）, SvmSimulateThread（读）
  保护: 通常在不同时间操作（训练/仿真不会同时进行）
```

### 5.2 串口设备文件的独占访问

```
/dev/ttyS1 → mySerial.comOpen() → 返回 fd
  访问: mySerialSendThread（只写）, mySerialRecvThread（只读）
  线程安全: Linux 内核保证全双工串口的读写不冲突

/dev/ttyS2 → mySerial.comOpen()
  访问: mySerial2SendThread（只写）, mySerial2RecvThread（只读）
```

### 5.3 SD 卡文件 I/O 竞争

```
/sdcard/cnf/*.json:
  访问者: GlobalFlow::saveGlobal()（写）, GlobalFlow::getGlobal()（读）
  保护: 无锁（保存和加载不在同一时刻发生）

/sdcard/logs/log.txt:
  访问者: Log4Qt Appender（写，单线程）
  保护: Log4Qt 内部线程安全

/sdcard/ts/statisticDataLog.db:
  访问者: MySqlite（写）, queryStatisticThread（读）
  保护: SQLite 自带文件锁（WAL 模式）
```

## 七、关键时序窗口

```
窗口 1: 串口未打开 (T=0 ~ T=70ms)
  此时如果硬件发来数据 → 丢失（无监听）
  解决: 硬件晚于上位机启动，或数据带重发机制

窗口 2: 串口已打开但加密未完成 (T=70ms ~ T=210ms)
  串口线程已在运行，但 GlobalFlow 尚未下发任何配置
  硬件处于"等待配置"状态

窗口 3: 加密完成但通信未握手 (T=210ms ~ T=300ms)
  updateStatusThread 的信号尚未 connect 到 ZKSort
  此时硬件状态变化 → updateStatusThread 发出信号 → 无人接收 → 丢失
  解决: 这段时间极短（100ms 级），且硬件通常不会在此窗口上报事件

窗口 4: 参数下发中 (T=300ms ~ T=2000ms)
  initSendAllParams 正在逐条下发配置
  用户尚不能操作（主页未显示）
  但串口通信和状态监控已全部就绪
```

## 八、资源消耗总览

| 资源 | 数量 | 说明 |
|------|------|------|
| 进程 | 1-5 个 | 主进程 + 按需子进程（train/unzip/wifi 脚本等） |
| 常驻线程 | 7-9 个 | 主线程 + 4 串口 + 1 状态 + 1-2 网络 + 0-2 MQTT |
| 临时线程 | 0-3 个 | 按需创建（查询/训练/测试/导出） |
| 文件描述符 | ~15-25 个 | 2 串口 + 2-4 TCP socket + 2 UDP + 日志文件 + 配置文件 + SQLite |
| 内存（估算） | ~50-80 MB | Qt 框架 30MB + 全局结构体 5MB + 页面延迟加载 15-40MB |
| 互斥锁 | 2 个 | myQueue 锁 + myQueue2 锁 |
